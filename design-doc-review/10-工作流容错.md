## 文档定位

M7 工作流体系的容错层设计：连线级 `fault` 字段声明五种失败策略，执行引擎按策略降级，图谱实时渲染容错状态。2026-07-19 由两份早期草稿（WORKFLOW_RESILIENCE.md 119 行 / WORKFLOW_RESILIENCE_IMPLEMENTATION.md 173 行）合并而成（9f72b6e），晚于"原子自愈系统"删除（5e308d4）仅 17 分钟，实为后者的精简替代。

## 核心决策清单

1. **五策略枚举**：retry / fallback / skip / timeout / abort，声明在**连线**而非节点上（`fault` 字段，含 strategy / max_retries / retry_intervals_ms / on_exhausted / fallback_value / timeout_seconds 六子字段，§三）。
2. **嵌套容错**：retry 耗尽后经 `on_exhausted` 降级到 fallback / skip / abort（默认 abort），仅 retry 可嵌套（§三）。
3. **零破坏兼容**：默认 abort = 现有行为，未声明 fault 的连线一切照旧（§十一）。
4. **执行语义**：拓扑序执行；输入侧 `resolve_fault`（源失败时按连线策略注入 fallback_value 或 SKIP）；节点异常落入"输出通道各自容错"（§五伪代码）。
5. **可观测性**：7 种节点状态环（pending/running/success/retrying/fallback/skipped/failed）+ 4 种连线状态 + 结构化容错日志（§六/§七）。
6. **UI**：点连线底部滑出容错面板（WorkflowMode.kt），GraphEdge.kt 渲染容错状态（§八/§九）。
7. **实施估算 5h**：改 CHANNEL_MODEL.md / workflow_executor.py / GraphEdge.kt / WorkflowMode.kt，增 tests/test_workflow_resilience.py，8 个测试场景（§九/§十/§十一）。

## 与代码现状的差距

1. **fault 字段零落地**：`fault` / `on_exhausted` / `fallback_value` / `retry_intervals` / `timeout_seconds` 在 mcp-yuanzi-bridge/ 全目录 grep 零匹配；workflow.py:19-116 校验器不识 fault（也不拒绝——009_workflows.sql 的 definition_json 是 opaque JSON 文本，字段可存但永不生效，无需新迁移）。
2. **执行引擎仅有"全局 retry + 硬编码 abort"**：engine.py:231-271 `while attempts <= max_retries`（默认 3，间隔 `retry_delay*attempts` 线性递增且 retry_delay 默认 0.0 即实际无间隔），最终失败即整流 FAILED——恰好是五策略中 retry（残版）+ abort 两项；无 fallback / skip / timeout；且 max_retries 是 run_workflow 调用级参数，不是定义级声明，无 per-channel/per-node 策略。
3. **timeout 无实现路径**：engine.py:39 `STATUS_TIMEOUT` 常量已定义但全文件零使用（STATUS_CANCELLED 同样空转）；handler 是同步函数调用（engine.py:139-141），Python 无法安全中断同步调用，伪代码 `run_with_timeout` 未给并发/隔离机制。
4. **skip 下游行为未定义**：`_gather_payload`（engine.py:104-133）对 direct/merge/split 一律 `payload.update()`，源缺失时该源贡献空 dict——skip 碰巧"能跑"（印证 01 组"合并无策略"）；但 map/transform 通道 skip 后 mapping 目标字段静默缺失，handler 收到缺参 payload 行为未定义；node_runs 无 skipped/fallback 状态（engine.py:249-268 仅 SUCCESS/FAILED），§七的容错日志无载体。
5. **示例工作流整体不可跑**：engine.py:142-145 注册原子抛 `ValueError("no live endpoint dispatch yet")`；示例六个原子全部来自传感层设计，`system.location` / `context-fusion` 在 mcp-yuanzi-bridge/ 与 base-atoms/ grep 零匹配——静态校验吃 6 个 "unknown base atom" warning（workflow.py:171-172），执行即 ValueError→FAILED。容错设计的主要受益对象（外部 API/传感器原子）恰恰全不可执行。
6. **Android 侧目标文件一不存在、一无实现**：`WorkflowMode.kt` 全仓不存在（find widgetmcp_src 仅命中 python 侧 workflow.py）；GraphEdge.kt grep fault/retry/abort/skip/fallback 零匹配，7 态环 / 4 态连线零渲染。
7. **文件名错位**：§九变更清单写 `workflow_executor.py`，实际执行器是 engine.py（全仓无 workflow_executor*）——合并稿沿用了被删实施方案的旧名。
8. **测试未建且无门禁护航**：tests/ 无 test_workflow_resilience.py；test_engine.py:68 `test_run_node_failure_retries_and_stops` 仅覆盖现有 retry+abort；CI 质量门禁失效（00 总览 P0-1，BUG-034），新增测试即使写了也无强制力。

## 风险与建议（按优先级）

1. 【高】裁决 skip 与 CONNECTION_RULES 规则 2 的矛盾：声明了 skip/fallback 的入边应在静态校验中豁免或降级为 warning，否则容错工作流"保存即被拒 / 运行时静默缺参"。
2. 【高】实施序列前置"注册原子可执行"（01 组缺口）：否则 fault 只服务 13 个基础原子，设计价值大打折扣。
3. 【高】修正文档硬错误：workflow_executor.py→engine.py；承认现有全局 max_retries 行为并定义"fault 声明 vs 调用级参数"的优先级；兑现 §九承诺，CHANNEL_MODEL.md 补 fault 字段定义。
4. 【中】澄清 retry 执行语义：统一连线声明与节点重试的对象；定义多入边策略冲突优先级；补副作用原子重试幂等警告（http-post 重试=重复提交）。

6. 【中】fallback_value 纳入类型匹配校验：至少对 map/transform 通道校验注入值满足目标必填输入。

8. 【低】示例改用可执行原子重写（如 system.http-get 失败 fallback），避免示例依赖未实现的传感层原子；测试落地前先修 CI（BUG-034）。
