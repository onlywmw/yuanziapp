## ADR_ATOM_MODEL.md

**定位**：原子生态基座决策——基础原子（内置不可删）与注册原子（可注册/发现/组合）两层模型统一。

**核心决策清单**
- 两层模型：基础原子单操作/无依赖/确定性；注册原子可依赖基础原子、需签名/版本/分类/合规。
- 统一接口：所有原子实现 `GET /health`、`GET /meta`、`POST /run`；handler 签名统一 `handler(data: dict) -> dict`，返回 `{"status": "success|error", ...}`。
- 运行环境：Linux（Android Termux 用户空间），Python 3.10+。
- 图谱可见性：基础原子可见/可搜/可作依赖，不可注册/不可删除，UI 特殊样式。
- 注册中心 `atom_registry` 新增 `is_base: bool` 字段；基础原子不签名、预分类、随系统升级。
- 基础原子建议清单 13 个（系统级 5 / 计算级 5 / 安全级 3）。

**关键契约或接口要点**
- 三端点 + 统一 handler 签名是全部原子的最低契约。
- 基础原子不可注册/不可删除需由 API 强制。

**文档与代码现状差距**
- 13 个基础原子已按建议清单落地：`base-atoms/`（file-dir、file-read、file-write、http-get、http-post、math-calc、string-split、string-match、json-parse、date-time、hash-digest、encrypt-aes、decrypt-aes），每个含 `server.py`（/health /meta /run，见 `base-atoms/http-get/server.py` 头注释）与 `core.py` 的 `handler(data)`（`base-atoms/http-get/core.py:49`）。✅
- `is_base` 字段**从未实现**：全仓库仅出现在本文档（grep 全仓仅 `docs/ADR_ATOM_MODEL.md:97`）。实际替代方案是 `system.` 命名空间保留（`mcp-yuanzi-bridge/registry.py:430` `RESERVED_PREFIXES`，提交 `system.*` 返回 `reserved_namespace`）+ META 中 `builtin: True`。决策被不同方式实现，ADR 未回填。
- ADR 背景所述"`atoms/` 下 5 个基础原子"已清空：`atoms/atom-*` 各目录仅剩 `__pycache__`，历史背景描述与现状无关（作为 ADR 背景可接受，但"两套模型互不连通"的问题陈述已过期）。
- 注册原子"需实现三端点"未强制执行：注册原子是远程 MCP 服务元数据，registry 仅存 `runtime.endpoint/health_url`，无三端点校验逻辑。
- 运行环境决策（Termux）已被 Chaquopy 路线取代：见跨文档矛盾。

## CHANNEL_MODEL.md

**定位**：通道（连线）模型——线是活的转换器，定义五种线类型与生命周期/视觉规范。

**核心决策清单**
- 线有三属性：类型（直通/映射/转换/合并/分流）、状态（静态/流动/阻塞/完成）、视觉（颜色=数据类型、粗细=数据量、动画=方向）。
- 五型规则：直通（输出⊆输入）、映射（字段名映射）、转换（类型转换函数）、合并（拼接/JSON合并/模板策略）、分流（全量/条件路由）。
- 每根线创建时声明 /meta：`channel_id`、`type`、`source_atom`、`target_atom`、`mapping`、`data_type`、`description`。
- 生命周期四态：静态（灰虚线）→流动（彩色动画）→阻塞（红闪）→完成（绿实线）。

**关键契约或接口要点**
- 通道类型枚举与各类型匹配/转换规则是工作流校验与执行的契约。

**文档与代码现状差距**
- 五型枚举已落地：`mcp-yuanzi-bridge/workflow.py:19` `CHANNEL_TYPES = {"direct","map","transform","merge","split"}`，校验逻辑覆盖 merge 多源、自环、未知端点（workflow.py:97-116）。✅
- 字段名不一致：文档用 `channel_id`/`source_atom`/`target_atom`，代码用 `id`/`source`/`target` 且指向**节点 id 而非 atom_id**（workflow.py:97-114）；`data_type`、`description` 未进校验。
- 转换线只实现了一半：文档要求"声明输入类型→输出类型+转换函数"，代码仅支持 `convert` 字典的两种硬编码模式 `str`/`json`（`engine.py:124-129`）。
- 合并线无策略：文档三种策略（拼接/JSON合并/模板）未实现，代码对 direct/merge/split 一律 `payload.update()` 字典覆盖合并（`engine.py:130-132`），后源覆盖前源，无拼接/模板。
- 分流线无条件路由：split 在引擎中与 direct 同处理（扇出整体传递），"条件路由"未实现。
- 生命周期四态与线的运行时状态**完全未实现**：`engine.py` 只记录 node_runs，通道无状态对象；静态/流动/阻塞/完成与动画视觉属 UI 层，本组未在 Python 侧发现对应实现。
- 注册原子不可执行：`_execute_node` 仅支持 `system.*` 基础原子，注册原子抛 `ValueError("no live endpoint dispatch yet")`（`engine.py:139-145`）——通道模型示例全用基础原子，与可执行范围一致，但"血液系统"目前只在基础原子间流动。

## 组内跨文档矛盾与不一致

1. **ADR §3 运行环境（Termux）vs 实现/其他设计**：ADR 定"Linux (Termux on Android)"为运行环境，但 `api.py` `start_server` 注释明确"Chaquopy 入口：Kotlin 侧 `py.getModule("api")` 调用"，`docs/DESIGN_CHAQUOPY_MIGRATION.md` 目标即"去掉 Termux 依赖"。ADR 决策已被后续决策覆盖，未标注"已修订"。
2. **SCHEMA_AUTHORITY 废弃 registry_meta vs M6/QA 文档**：代码已无 registry_meta，但 `docs/DESIGN_M6_SECURITY.md:76,149` 与 `qa/tests/docs/BUG-025-ACCEPTANCE-CRITERIA.md:19,54,60` 仍以 registry_meta 为 token 存储设计——文档间直接冲突。
3. **INTERFACE_CONTRACTS §1.3 状态机 vs §1.9 probe 副作用**：契约自身引用了 unreachable 状态（§1.9）却未把它列入 §1.3 的合法状态/转换表，文档内部即不一致；代码六态机证明 §1.3 才是过期的一方。
4. **ARCHITECTURE_OVERVIEW 自述"15 份文档" vs 实际列出 18 份 vs docs/ 实有 35 份**：三重数字互相矛盾，索引失去权威性的同时拉低其指向的 4 份基座文档的可信度。
