## 文档定位

提出在 13 个"工具原子"之外补四类基础原子：感知（sensor）/ 融合（fusion）/ 决策（rule）/ 执行（actuator），共 12 个新 `system.*` 原子，基础原子从 13 扩到 25。
无状态头、无作者、无日期；与同日文档 DESIGN_ATOM_V2_CLASSIFICATION.md（`design-ready`，Arch，2026-07-19）内容几乎完全重叠，后者带 schema 变更与实施计划，是同一决策的"正式版"。

## 核心决策清单

- 新增 12 个 `system.*` 原子：感知 6（location / camera / weather / device / clock / biometric[未来]）、融合 1（context-fusion）、决策 1（rule-engine）、执行 4（music-player / notification / display / vibrate）
- 分类归属：全部挂在 `ARCHITECTURE.type` 下新增枚举值 sensor / fusion / actuator；`rule` 复用 v1 已有枚举值（`atom-registry-schema.json:90` 已含 `"rule"`）
- 感知原子多为"输入： 无 — 主动采集"，带**事件触发语义**（进入/离开地理围栏、天气变化、设备连接）——即文末"粒子触发粒子"：感知→决策→执行自动级联，非人手动跑工作流
- 融合原子把多路感知合成单情境 `{situation, mood, tags[], confidence}`；决策原子做 IF-THEN 规则匹配输出 `{matched_rule, action, params}`
- 典型场景链：location+weather+camera+device+clock → context-fusion → rule-engine → music-player + notification（雨天咖啡厅播《Stan》）

## 与既有文档的兼容性

**矛盾点：**

1. **与 DESIGN_ATOM_V2_CLASSIFICATION.md 同质重复（首要矛盾）**：两份文档定义完全相同的 12 个原子、相同的雨天咖啡厅场景。SENSOR_LAYER 无状态头；V2 有 `design-ready` 状态头 + schema 变更（§三）+ 四阶段实施计划（§五）。字段级目前一致（weather 输入 `{location}`、rule-engine 输出三字段均同），但双份规格必然漂移，需裁决唯一权威源。
2. **"触发"语义与连线规则体系不兼容（隐性矛盾）**：ATOM_CONNECTION_RULES / WORKFLOW_CONNECTION_RULES 只定义静态依赖与 direct/map/transform/merge/split 五类数据通道（`workflow.py:18` CHANNEL_TYPES），全文无"事件触发"概念；SENSOR_LAYER 的地理围栏/天气变化触发是**推模式**，与规则文档的拉模式 DAG 是两种执行范式，连线规则文档未为新范式留任何位置。
3. **世界效应环未纳入危险链**：既有五类危险链（ATOM_CONNECTION_RULES §三）全是数据流形态；actuator→sensor 的真实世界反馈自激环（如 vibrate → device 感知变化 → rule-engine → 再 vibrate）是新危险类别，两份连线规则文档均未覆盖。
4. **system.clock 与既有 system.date-time 功能重叠**：13 基础原子已有 date-time（`base-atoms/date-time/core.py:15-28`，now/format/diff + 时区），新 clock 输出 `{hour, day, is_weekend, season}` 与之部分重复，文档无分工或合并说明。
5. **weather 的分类张力**：weather 输入 `{location}` 主动查询 OpenWeather（V2 §感知类注明来源），本质是 external 型 API 调用而非设备感知，归入 sensor 与 v1 分类语义（external=外部调用）有张力，未讨论。

**未发现矛盾的部分：**

- 层级矩阵兼容：12 个新原子全是 `system.*`，其内部 sensor→fusion→rule→actuator 链均属"基础→基础 ✅"，不触碰"基础不得依赖注册"禁令（ATOM_CONNECTION_RULES §一）。
- 命名空间兼容：`system.*` 是保留前缀（`mcp-yuanzi-bridge/registry/schema.py:19` RESERVED_PREFIXES），新原子只能作内置基础原子，走不了注册路径——与 REGISTERED_ATOM_RULES 无冲突。
- schema 层面无硬阻塞：`architecture.type` 枚举扩 3 个值即可；`classification.category` 是自由字符串（`atom-registry-schema.json:155`），V2 §三的 category 新增无需 schema 改动。
- `rule` 类型表述一致：SENSOR_LAYER §一把 rule 列为"当前已有"，与 v1 schema 枚举（`atom-registry-schema.json:90`）吻合。

## 与代码现状的差距（带证据）

- **12 个原子零实现**：`base-atoms/` 仅 13 个既有目录；全仓 grep `system.(location|camera|weather|device|clock|context-fusion|rule-engine|music-player|notification|display|vibrate)` 在代码侧零命中，仅出现在三份文档（ATOM_SENSOR_LAYER / DESIGN_ATOM_V2_CLASSIFICATION / DESIGN_WORKFLOW_RESILIENCE）。
- **registry 无任何注册/类型路径**：`mcp-yuanzi-bridge` 内 grep `sensor|actuator|context-fusion|rule-engine|music-player` 零命中；61 个注册原子 `architecture.type` 全部是 `mcp-server`（ATOM_REGISTRY_LEDGER.json 统计）。
- **事件触发执行模型零支持**：`engine.py` grep `trigger|event|subscribe|webhook|poll|listen` 零命中，引擎只有同步拉取执行 + 单次 timeout=10s 重试 3 次；"粒子触发粒子"需要的触发调度器/事件通道不存在任何代码与设计文档（连线规则文档也无）。
- **schema 枚举未扩展**：`atom-registry-schema.json:90` 枚举仍为 v1 八值，无 sensor/fusion/actuator（V2 §三阶段 1 第 1 步即此事，未做）。
- **静态检查表未覆盖**：`workflow.py:24-38` BASE_ATOM_IO 硬编码 13 原子；新 12 原子若落地不入表，将落入 `workflow.py:173`"注册原子的 I/O 由 meta 决定，跳过静态检查"的同一跳过路径。
- **JSON Schema 校验不在 live 注册路径**：`validate_atom`（Draft7Validator）只存在于批量脚本 `register_mcp_atoms.py:356` 与测试；live 提交入口 `registry/core.py:155-230` submit_atom 只做命名空间/签名/content_hash 去重，无 schema 校验——即使扩了枚举，注册路径也不会强制执行。
- **Android 侧零实现**：`widgetmcp_src` grep `context-fusion|rule-engine|system.location|geofence` 零命中；V2 阶段 4（APK 感知层）未启动。
- **已被下游设计前置引用**：DESIGN_WORKFLOW_RESILIENCE.md（`design-ready`）的容错通道示例已使用 system.weather/location/camera/device/context-fusion（行 132-192）——未存在的原子已进入另一份设计的契约示例。

## 与既有审计结论的关系

- **加剧 · "注册原子静态检查被跳过"（02 组发现）**：跳过逻辑按"不在 BASE_ATOM_IO 即跳过"工作，12 个新基础原子落地即自动加入跳过面，静态类型检查的覆盖缺口从 61 个注册原子扩大到全部新基础原子。
- **加剧 · "两条禁令无代码牙齿"（02 组发现）**：新原子扩大了"system.* 不得依赖注册原子"的适用面而禁令仍无校验；且本次新发现 live 注册路径（`registry/core.py:155`）连 JSON Schema 校验都未接入——枚举扩展也牙齿落空，02 组"注册路径无强制"的结论比原报告记录的更深一层。
- **新增 · 执行范式缺口**：既有审计未发现"事件触发"维度——SENSOR_LAYER 引入的推模式触发（地理围栏/天气/设备事件）在 engine.py、workflow.py、两份连线规则文档中均无对应物，是比"静态检查跳过"更底层的缺口：即使原子实现完毕也无法被自动触发，只能手动跑。
- **新增 · 文档孪生**：SENSOR_LAYER（无状态头）与 V2_CLASSIFICATION（design-ready）同质并存，属于 06 组"文档治理欠账"家族的新成员，且下游 RESILIENCE 文档已前置依赖，失真会级联。
- **印证 · 总览"文档体系系统性失真"**：方向相反——此处是文档大幅超前代码（纯构想，零实现），与 M4~M7 的"代码超前文档"互为镜像，同样造成文档可信度折损。
- **部分缓解 · 命名空间治理**：新原子全部走 `system.*` 内置路径（RESERVED_PREFIXES 强制），绕开了注册路径的全部校验缺失——若真作内置原子实现，02 组的注册侧牙齿问题对其不适用；但代价是内置原子只能硬编码进 BASE_ATOM_IO 才有静态检查，回到"加剧"项。

## 风险与建议（按优先级）

1. 【高】裁决文档孪生：以 V2_CLASSIFICATION 为权威源，给 ATOM_SENSOR_LAYER 补状态头并标 superseded（或反向合并），12 原子规格同步回写 BASE_ATOMS_SPEC（V2 阶段 1 已自认此步）。
2. 【高】先设计事件触发执行模型（触发调度器 + 事件通道类型 + 与 DAG 引擎的衔接），再实现原子——否则 12 原子落地即用户不可达死代码，重蹈总览 P0-3"M8 三件套未接线"覆辙。
3. 【高】连线规则补"世界效应环"类别：actuator→sensor 反馈自激环（震动→设备感知→规则→再震动）需进 ATOM_CONNECTION_RULES 危险链清单并在 workflow.py 配校验牙齿。
4. 【中】实现时同步把 12 原子 I/O 写入 `workflow.py` BASE_ATOM_IO 表，避免静态检查跳过面扩大；顺带修正 02 组已发现的 I/O 表口径差。
5. 【中】把 `validate_atom`（register_mcp_atoms.py:356）接入 live 注册路径 `registry/core.py:155` submit_atom，再谈 type 枚举扩展——否则枚举只是文档装饰。
6. 【中】裁决 system.clock 与既有 system.date-time 的分工（合并/扩展 date-time 输出 is_weekend/season，或明确 clock 为事件触发封装），消除功能重叠。
7. 【低】weather 重新归类或论证：按 v1 语义它更接近 external（API 查询），归 sensor 需一句分类标准说明（"感知=情境输入"）。
8. 【低】rule-engine 的"规则存储于 atom_registry 中 type=rule 的原子"（V2 §决策类）意味着"规则即原子"，其版本管理/审核流程与 REGISTERED_ATOM_RULES 的关系需补一节设计。
9. 【低】DESIGN_WORKFLOW_RESILIENCE 示例前置引用未存在原子，应加"依赖感知原子层落地"的前置说明，避免容错设计契约悬空。
