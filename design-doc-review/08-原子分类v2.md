# 增量审计 08 · DESIGN_ATOM_V2_CLASSIFICATION.md（原子分类 v2）

> 审计时间：2026-07-19 ｜ 文档状态：`📐 design-ready`（2026-07-19，Arch，提交 b1771e0）｜ 性质：增量文档，不在此前 35 份分组审计范围内

## 文档定位

将基础原子从 13 个"工具原子"扩展为五类 25 原子（工具 13 / 感知 6 / 融合 1 / 决策 1 / 执行 4），新增 12 个 `system.*` 原子的 I/O 规格、schema 枚举扩展方案与四阶段实施计划。本质是把先一日提交的 ATOM_SENSOR_LAYER.md（5207f86）正式化为分类体系 + 落地路线图。

## 核心决策清单

- 五类分类法：工具（既有 13）/ 感知 sensor（location、camera、weather、device、clock、biometric）/ 融合 fusion（context-fusion）/ 决策 rule（rule-engine）/ 执行 actuator（music-player、notification、display、vibrate）。
- 12 个新原子全部使用 `system.*` 命名，定位为**基础原子**（内置），不走注册流程；biometric 标注"未来实现"。
- Schema 变更：`atom-registry-schema.json` 的 `architecture.type` 枚举新增 sensor/fusion/actuator 三值（rule 复用既有枚举值）；`classification.category` 新增四个类目。
- 数据流契约：感知 → context-fusion（输出 situation/mood/tags/confidence）→ rule-engine（规则存于 atom_registry 中 type=rule 的原子）→ 执行原子，场景示例"雨天咖啡厅 → 播放《Stan》"。
- 实施四阶段：Schema（30min）→ 12 个 base-atoms 目录（core.py+server.py+Dockerfile，1 天）→ 工作流引擎 rule-engine/fusion（1 天）→ APK 感知层采集（1 天）。
- 注册原子侧不变：mcp.* 领域原子 61+ 保持，终端原子（商品/服务/作品）归为 M7。

2. **与 ADR_ATOM_MODEL 是"扩展"而非"替代"，但未声明关系**：ADR 的两层模型（基础/注册）与 v2 不冲突——v2 是在"基础原子"层内部再分五类，可并存。但 ADR/BASE_ATOMS_SPEC 的三级分类（系统级 5/计算级 5/安全级 3）被 v2 整体并入"工具 (13)"，三级口径被静默搁置，v2 只字未提废止还是并存；ADR 的 `is_base` 字段、三端点契约对新五类是否适用也未说明。

## 与代码现状的差距（全部未实现，证据如下）

- **Schema 阶段（自称 30min）未做**：`atom-registry-schema.json:90` type 枚举仍为 8 值 `["function","rule","external","agent","mcp-server","data","schema","gateway"]`，无 sensor/fusion/actuator；`classification.category` 在第 155 行是自由 string（无枚举），v2 所称"category 新增"无任何 schema 约束落地。
- **12 个新原子目录零存在**：`base-atoms/` 实测仅 13 个工具原子目录 + tests + README（file-dir、file-read、file-write、http-get、http-post、math-calc、string-split、string-match、json-parse、date-time、hash-digest、encrypt-aes、decrypt-aes），无 location/camera/weather/device/clock/biometric/context-fusion/rule-engine/music-player/notification/display/vibrate。
- **测试文件未建**：`base-atoms/tests/` 仅有 `test_base_atoms.py`，无 v2 要求的 `test_sensor_atoms.py`/`test_sensors.py`；全仓 grep `sensor|fusion|actuator` 在代码与 schema 中零命中（仅命中 4 份 docs）。
- **命名空间保护反而构成实现约束（v2 未提及）**：`mcp-yuanzi-bridge/registry/schema.py:19` `RESERVED_PREFIXES = ("system.", "yuanzi.")`，`registry/core.py:164` 提交拦截返回 `reserved_namespace`——12 个新原子**无法经注册 API 入库**，只能走内置硬编码路线；v2 未说明内置原子如何在 registry/图谱中可见（现有 13 个工具原子同样不在 registry.db 中，该问题为既有 debt 的延续）。
- **执行与校验链路未预留接口**：工作流引擎 `_execute_node` 仅支持 13 个 `system.*` 基础原子（见 01 报告引 engine.py），`workflow.py` 的 `BASE_ATOM_IO` 类型表只有 13 条——v2 实施计划只有"新增目录"，未列 engine 分发、workflow I/O 表、resolve_dependencies 等必须同步改动的集成点，四阶段计划不完整。
- **平台张力未解**：v2 阶段 2 要求新原子沿用 core.py+server.py+Dockerfile 微服务模式，但 location/camera/device 数据来源是 Android LocationManager/CameraX/BluetoothAdapter（阶段 4 才做 APK 采集）——Python 侧 server.py 拿不到这些数据，"感知原子 = base-atoms 微服务"的形态与"APK 系统 API 采集"之间存在未设计的桥接层（Kotlin→Python 数据注入？），文档内部两阶段自相矛盾。
- **M7 口径滞后**：v2 把"终端原子 — 商品/服务/作品 (M7)"写作规划项，但 M7 marketplace/workflow 已落地（见 04 组报告），描述与现实脱节。
- 既有 13+61 基数属实：`base-atoms/` 13 目录、`mcp-yuanzi-bridge/mcp_atoms.json` 实测 61 条，v2 的 v1 基线描述准确 ✅。

## 与既有审计结论的关系

- **印证 01 组"ADR is_base 从未实现、被 system. 命名空间替代"**：v2 新增 12 原子同样不引入 is_base，全部走 `system.*` 内置路线，等于再一次以设计文档形式确认 ADR 的 is_base 决策已死亡，ADR 回填的紧迫性 +1。
- **印证 02 组"基础原子层与文档高度吻合"**：v2 完整保留 13 个工具原子及其规格，未与 BASE_ATOMS_SPEC 的 I/O 定义冲突。
- **加剧"计数多套不一致"问题**：原子计数现已有三套口径并存——代码 13 基础 + 61 注册、BASE_ATOMS_SPEC/OVERVIEW 条目 13、v2 25——且 OVERVIEW 索引同时背书 13 与 25，继文档计数（15/18/35/36）之后再添一处数字失真。
- **新增"同主题双文档"治理问题**：ATOM_SENSOR_LAYER 与 DESIGN_ATOM_V2_CLASSIFICATION 内容 ~90% 重复、均无 supersede 标注，属此前 35 份审计中未出现过的重复类型（此前的矛盾是"内容冲突"，此次是"内容复制"）。
- **新增"实施计划覆盖不全"模式**：v2 四阶段计划遗漏 engine/workflow 集成点，与 03 组"混音台文档无视执行链路现状"同构——设计文档继续以绿地方式规划，不盘点现有代码挂载点。
- **缓解项（小幅）**：v2 明确 rule 复用既有 schema 枚举值、只新增三个枚举值，范围克制；且把 biometric 标为"未来实现"，范围切割清晰，未夸大落地口径。
