# 02 · 原子规格组 — 设计文档精读审计报告

## 组概述

覆盖 7 份文档：基础原子规格、注册规则、灵魂模型（规格+实施）、可见性模型、连线规则（原子级+工作流级）。
总体结论：基础原子层（base-atoms/）与文档高度吻合；连线规则在工作流引擎（workflow.py）中大部分落地；
灵魂模型两份文档互相矛盾且完全未实现；可见性模型未实现且其"零改动"主张不成立；注册时循环检测与层级防护规则在代码中缺失。

## 1. BASE_ATOMS_SPEC.md

**一句话定位**：13 个 system.* 内置原子的 I/O 规格、目录结构标准与旧 atoms/ 迁移清单。

**核心决策清单**
- 13 个基础原子分三级：系统级 5（file-dir/file-read/file-write/http-get/http-post）、计算级 5（math-calc/string-split/string-match/json-parse/date-time）、安全级 3（hash-digest/encrypt-aes/decrypt-aes）
- 统一接口：`/health` `/meta` `/run`，`handler(data)` → `{status, data/message}`
- 安全约束：文件限白名单目录 + max 5MB；HTTP 限 http/https 公网 + max 100KB + timeout 30s
- 目录标准：core.py（唯一业务逻辑）+ server.py + requirements.txt + Dockerfile
- 旧 atoms/ 5 目录 + tests 列入删除清单

**关键契约要点**
- 每个原子原子级 I/O 字段表（如 file-read 输入 `{path, mode, encoding}`，约束 max 5MB）
- 旧→新映射表（atom-math-sum → math-calc 等 5 条）

**文档与代码现状差距**
- 13 个原子全部存在且结构（core.py/server.py/requirements.txt/Dockerfile）与文档一致（`base-atoms/`）；/health /meta /run 三端点落实（`base-atoms/file-read/server.py:70-78`）
- file-read 白名单 + 5MB 落实（`base-atoms/file-read/core.py:26-40`）；http-get scheme 白名单/内网拒绝/100KB/30s 落实（`base-atoms/http-get/core.py:9-65`）
- **删除清单未执行**：`atoms/atom-math-sum`、`atom-string-split`、`atom-file-read`、`atom-http-get`、`atom-template`、`atoms/tests/` 全部仍存在 —— 文档声明的迁移只完成了一半
- math-calc 文档只写"表达式计算"，代码注释明确"AST 安全求值，不使用 eval()"（`base-atoms/math-calc/core.py:6`），文档未体现这一安全决策

## 2. REGISTERED_ATOM_RULES.md

**一句话定位**：注册原子的最低规范——必填字段、author 神圣性、注册流程与禁止项。

**核心决策清单**
- 必填：atom_id / author / name / description / purpose.summary / purpose.functions
- author 不可为空、不可伪造、一经注册不可更改；位于 `ownership.author`
- 不强制声明基础原子依赖、不限语言、不限部署方式
- 禁止重复注册（SHA-256 签名去重）；probe 机制持续验证可达性

**关键契约要点**
- ownership 子对象结构（author 必填，maintainer/organization/license/source_url 可选）
- 注册流程：提交 → 自动签名 → 自动分类 → 审核 → 注册成功

**文档与代码现状差距**
- **description 必填与 schema 不符**：`atom-registry-schema.json` 顶层 required 为 `[atom_id, name, purpose, architecture, ownership, lifecycle, signature]`，无 description；嵌套 required 仅 `purpose.summary/functions`、`ownership.author` —— 文档的 description 必填无强制
- author 必填已落实（schema ownership.required + 61 个 ledger 原子 author 全部非空，`mcp-yuanzi-bridge/ATOM_REGISTRY_LEDGER.json`）
- **"author 一经注册不可更改"无代码强制**：`mcp-yuanzi-bridge/registry.py` 更新路径（update_atom，~364 行）无任何 author 不变性校验
- 签名去重落实（`registry.py:438-449` duplicate_signature 检查）；probe 机制存在（`mcp-yuanzi-bridge/probe_atoms.py`）
- "61 个原子"与 ledger 一致（61 个，全部 registered 状态）

## 3. ATOM_SOUL_MODEL.md

**一句话定位**：在 classification 下新增 soul 子对象（风格/受众/基调/品质/场景/叙事）的产品规格。

**核心决策清单**
- soul 六字段：style[]、audience[]、mood、quality、use_case[]、narrative（10-200 字）
- **风格标签池"开放, 不限制"**（§2.1 原文）
- 填写义务：基础原子 system 预填、注册原子可选、终端原子必填
- 验证：风格/受众各最多 3 个；叙事 10-200 字 + 人工审核
- soul 全局可选，"不破坏现有 277 测试和 61 原子"（§五）

**关键契约要点**
- `classification.soul` JSON 结构（§五）
- 对 M5 搜索的加权影响：soul 匹配 > 纯功能匹配；同 style/mood/quality/作者推荐

**文档与代码现状差距**
- **完全未实现**：`atom-registry-schema.json` classification 仅 `{category, domain, tags, maturity}`，无 soul；全仓库 grep `soul|validate_soul` 在 mcp-yuanzi-bridge 零命中
- "277 测试"口径无法在代码复现：可点名的 pytest 测试函数为 202（`mcp-yuanzi-bridge/tests/`）+ 23（`base-atoms/tests/test_base_atoms.py`）≈ 225
- 终端原子必填 soul 的规则无任何校验入口（注册校验尚无 validate_atom 级别的灵魂检查）

## 4. SOUL_MODEL_IMPLEMENTATION.md

**一句话定位**：灵魂模型的落地方案——schema 变更、迁移 011、validate_soul、搜索权重、UI 渲染，号称 ~2 小时零破坏。

**核心决策清单**
- schema 加 soul（maxItems 3、narrative minLength 10/maxLength 200）
- `migrations/011_add_soul.sql` 为占位迁移（无 DDL 变更，soul 存 classification_json TEXT 列）
- registry.py 新增 validate_soul：SOUL_STYLES/AUDIENCES/MOODS/QUALITIES **封闭枚举**，未知标签报错
- 搜索权重：功能 0.5 / 灵魂 0.3 / 叙事 0.2，`POST /search` 加 `soul_match` 参数
- DetailPanel.kt / MarketCard.kt 加渲染；tests/test_soul.py 四个用例

**关键契约要点**
- validate_soul 函数签名与四个标签池常量（§3.1，可直接作实现蓝本）
- 向后兼容矩阵：旧原子 soul=null → API 返回 null → UI 不渲染 → 纯功能搜索

**文档与代码现状差距**
- **迁移编号冲突**：文档指定 `migrations/011_add_soul.sql`，但实际 `mcp-yuanzi-bridge/migrations/011_federation_peers.sql` 已占用 011；且迁移目录在 `mcp-yuanzi-bridge/migrations/`，非文档暗示的仓库根 `migrations/`
- 全部条目（schema 字段、validate_soul、soul_match 搜索参数、Kotlin 渲染、test_soul.py）均未实现；widgetmcp_src 中 "SOUL" 仅是 ColorScheme 配色枚举名（`graph/templates/GraphTemplate.kt:91`），与灵魂模型无关
- 与 ATOM_SOUL_MODEL 的核心矛盾见"跨文档矛盾"第 1 条

## 5. ATOM_VISIBILITY_MODEL.md

**一句话定位**：不设标签，用 visibility_score(atom) 浮点分数决定原子在图谱中的展示/折叠/隐藏/不可见四层渲染。

**核心决策清单**
- 评分公式：narrative +3.0 / style +1.5 / audience +1.0 / mood +0.5 / quality +0.5 / 评分≥10条 +1.0 / 均分≥4.0 +0.5；system.* 强制归零；值域 0~7.5
- 分数映射：≥4.0 展示层、1.0~3.9 折叠层、<1.0 隐藏层、=0 不可见
- 渲染规则：默认只渲染 ≥4.0；"显示全部"渲染 ≥1.0；搜索无视分数；双击显示低分邻居
- 声称"零改动"：不新增 schema 字段/列/API 参数，"只加一个函数 visibility_score"

**关键契约要点**
- visibility_score(atom) → float 的完整伪代码（§二，可直接实现）

**文档与代码现状差距**
- **未实现**：全仓库无 visibility_score；Kotlin 图谱侧无对应渲染层级逻辑
- **"零改动"不成立**：评分依赖 `atom["ratings"]`，但 schema 无 ratings 属性，评分数据在 `atom_reviews` 表（`mcp-yuanzi-bridge/migrations/010_atom_reviews.sql`，rating 1-5）——实现必须做聚合查询，且 GraphEngine/前端渲染分层是实质改动，远超"一个函数"
- 依赖前提 soul 字段本身未实现（见上两节），该模型当前无可计算输入

## 6. ATOM_CONNECTION_RULES.md

**一句话定位**：原子间静态依赖的层级规则（基础/注册两层）+ 五类危险链警告 + 运行时约束。

**核心决策清单**
- 层级矩阵：基础→基础 ✅、基础→注册 ❌、注册→基础 ✅、注册→注册 ⚠️ 有条件
- 五类危险链（入站→写文件、读文件→出站、解密→出站、外网输入→执行、循环写入）：**警告但不强制阻断**
- 注册时自动运行 resolve_dependencies() 检测循环，**有循环 → 注册被拒**
- 运行时约束：不可自调用、链式总超时 ≤30s、深度 ≤5 层、内存调用计数防死循环

**关键契约要点**
- 允许/禁止/警告三组连线速查示例（§四）

**文档与代码现状差距**
- **注册时循环检测未接入**：`resolve_dependencies`（`registry.py:1060`，含 cycles 检测）仅暴露为查询端点 `GET /atoms/{id}/dependencies`（`api.py:224`）；submit_atom 注册路径不调用它 —— 循环依赖可注册成功，文档"注册被拒"不成立
- **层级规则无强制**：注册/更新路径无"system.* 原子 dependencies 不得指向注册原子"的校验（`registry.py` submit/update 无 dependencies 内容检查）
- **运行时约束三项未实现**：`engine.py` 无深度限制、无调用计数；单次调用 timeout=10s + 重试 3 次（`engine.py:64,152`），无"链式总超时 30s"概念
- 五类危险链中前四类已在工作流层落地为警告（`workflow.py:41-64` _DANGER_PATTERNS）
- 3.4 节假设 "math-calc (eval 模式)"，但实现是 AST 安全求值无 eval 模式（`base-atoms/math-calc/core.py:6`）——危险链前提在当前实现中不成立，规则表述过时

## 7. WORKFLOW_CONNECTION_RULES.md

**一句话定位**：工作流编排层的连线校验——类型匹配、必填覆盖、自循环/无意义链/读写环五规则 + 管线角色分类。

**核心决策清单**
- 规则 1 类型匹配；规则 2 缺必填参数=无效；规则 3 自循环禁止；规则 4 同类重复=警告；规则 5 写操作不成环（❌ 无效）
- 13 基础原子的输入/输出类型表（§一）
- 管线角色：源/处理/汇三分
- 验证流程 5 步，失败返回"哪个原子→哪个原子，缺少什么"
- 前提：每个原子 /meta 必须声明输入/输出 schema

**关键契约要点**
- BASE_ATOM_IO 类型表；有效/无效连线速查（§三/§四）

**文档与代码现状差距**
- **落地良好**：`mcp-yuanzi-bridge/workflow.py` validate_workflow 实现通道类型校验、自循环错误、DAG 无环（硬错误）、必填参数覆盖检查（含 param 节点）、危险链警告、direct 连线 I/O 无交集警告（`workflow.py:71-237`）
- **I/O 表小差异**：文档 string-split 输入列 `text, delimiter`（未标可选），代码必填仅 `{text}`（`workflow.py:26`）；file-dir 输出文档写 `entries[]`，代码为 `{entries, count}`
- **规则 2 与代码的口径差**：文档把 file-read→encrypt-aes（缺 key）列为 ❌ 无效，代码用 SYSTEM_PROVIDED_INPUTS 放行 key/headers/timeout 等 6 个参数（`workflow.py:38`）——代码采用的是文档 §三速查示例 2"key 来自系统配置"的宽松口径，文档自身两处矛盾（见跨文档矛盾第 3 条）
- **注册原子静态检查被跳过**：`workflow.py:173` 注释"注册原子的 I/O 由 meta 决定，跳过静态检查"——类型匹配只对 13 个基础原子生效，文档未声明这一边界
- /meta 声明 I/O 的前提成立（`base-atoms/file-read/server.py:23-27` input_schema/output_schema）

## 组内跨文档矛盾与不一致

1. **灵魂标签池开放 vs 封闭（直接冲突）**：ATOM_SOUL_MODEL §2.1 明写"风格标签池 （开放， 不限制）"；SOUL_MODEL_IMPLEMENTATION §3.1 的 validate_soul 用封闭枚举 SOUL_STYLES 拒绝一切未知标签。实施文档推翻了规格文档的核心产品决策，且无任何说明。
2. **读写环的处置三处不一**：ATOM_CONNECTION_RULES §3.5 列为"警告，不强制阻断"；WORKFLOW_CONNECTION_RULES 规则 5 标 ❌ 无效；workflow.py 实现为硬错误（`cycle detected`）。三份口径需统一（代码最严）。
3. **WORKFLOW_CONNECTION_RULES 文档内部矛盾**：规则 2 示例与 §四无效速查均把 `file-read → encrypt-aes`（缺 key）判为 ❌；§三有效速查示例 2 却把同一链列为 ✅（"key 来自系统配置"）。代码跟随后者。
4. **ATOM_VISIBILITY_MODEL 边界自相矛盾**：§二定义 score=0 为"不可见（system.* 永不绘制）"、score<1.0 为"隐藏层"；但 §二成长过程示例写"刚注册 soul 为空： score=0 → 隐藏层"。score=0 同时落入两档，新注册原子的归宿不明。
5. **可见性模型 vs 现实数据模型**：ATOM_VISIBILITY_MODEL 声称"不新增字段、只加一个函数"，但其公式依赖的 soul（未实现）与 ratings（存于 atom_reviews 表而非 atom 对象）都不在 atom dict 中，"零改动"承诺与 ATOM_SOUL_MODEL/SOUL_MODEL_IMPLEMENTATION 的新增字段计划直接冲突。
6. **迁移编号冲突**：SOUL_MODEL_IMPLEMENTATION 指定的 `011_add_soul.sql` 编号已被 `mcp-yuanzi-bridge/migrations/011_federation_peers.sql` 占用。
7. **注册口径张力（非硬冲突）**：REGISTERED_ATOM_RULES"不强制声明对基础原子的依赖"与 ATOM_CONNECTION_RULES"register 时自动 resolve_dependencies 检测循环、循环即拒"之间存在口径落差——若依赖可不声明，注册时循环检测形同虚设；代码现状（注册不检测）实际站在了注册规则一边。
8. **危险链前提过时**：ATOM_CONNECTION_RULES §3.4 针对 "math-calc eval 模式" 的代码注入警告，与 BASE_ATOMS_SPEC 实现（AST 安全求值、无 eval）不符；workflow.py 仍保留该警告模式，属防御性冗余。

## 风险与建议（按优先级）

1. 【高】注册路径接入 resolve_dependencies 循环检测 + "system.* 不得依赖注册原子"层级校验，否则 ATOM_CONNECTION_RULES 的两条核心禁令均无牙齿（registry.py submit/update 路径）。
2. 【高】灵魂模型动手前先裁决"标签池开放 vs 封闭"矛盾，并把迁移编号改为 014_add_soul.sql、路径修正为 mcp-yuanzi-bridge/migrations/。
3. 【高】可见性模型"零改动"承诺需改写：明确依赖 soul 落地 + atom_reviews 聚合 + GraphEngine 渲染分层三项前置工程，排期在灵魂模型之后。
4. 【中】统一读写环口径（建议采用 workflow.py 的硬错误口径，回改 ATOM_CONNECTION_RULES §3.5 为"禁止"）。
5. 【中】消解 WORKFLOW_CONNECTION_RULES 规则 2 与速查示例 2 的内部矛盾：明确定义 SYSTEM_PROVIDED_INPUTS 白名单参数（key/headers/timeout/encoding/mode/algorithm）为规则 2 的正式例外。
6. 【中】执行或撤回 BASE_ATOMS_SPEC 删除清单：旧 atoms/ 五目录与 tests 仍在仓库中，迁移处于半完成态。
7. 【中】为 author 不可更改加代码强制（registry.py update 路径比对原 author），或修改 REGISTERED_ATOM_RULES 表述。
8. 【低】schema 与注册规则对齐：决定 description 是否真必填（改 schema required 或改文档）。
9. 【低】修正文档数字口径："277 测试"与实际可收集测试数（约 225 个 pytest 用例）不符；ATOM_CONNECTION_RULES §3.4 "eval 模式"表述更新为 AST 求值后的真实风险面。
10. 【低】WORKFLOW_CONNECTION_RULES §一 I/O 表与 workflow.py BASE_ATOM_IO 对齐（string-split delimiter 可选、file-dir 输出含 count），并补写"注册原子静态检查跳过"的边界说明。
