## 文档定位

新增第 13 个"感知层家族"`system.*` 原子 `system.ai`：本地自然语言意图理解（query → intent/params/matched_atoms/matched_workflows），定位为"补在 rule-engine 前面"的前置理解层，A（ONNX 小模型）+ B（规则关键词兜底）双轨。
无状态头、无作者、无日期（与 ATOM_SENSOR_LAYER 同一裸文档形态）；未被 ARCHITECTURE_OVERVIEW.md 索引（总目仍 36 份，grep `AI_INTENT` 零命中）。

## 核心决策清单

1. **原子定义**：`atom_id: system.ai`，类型标注 `intelligence`，输入 `{query, context}`，输出 `{intent, params, confidence, matched_atoms[], matched_workflows[]}`；运行形态"Chaquopy 内嵌 Python → ONNX Runtime → 小模型"，延迟 <200ms，模型 ~50MB。
2. **三方案选型**：A = distilbert 意图分类器 ONNX 导出（~45MB，<100ms，打包进 APK 或首启下载）；B = 规则+关键词零模型兜底（<1ms）；C = 用户本地 Claude CLI（2-5s，文档自认 Node.js 环境 APK 内不一定有）。推荐 A+B 双轨。
3. **意图→原子映射表**：play_music→system.music-player、check_weather→system.weather、find_place→system.location、send_message→system.notification、create_note→system.file-write、recommend→search+工作流推荐。
4. **三种工作流关系**：AI 直接触发工作流 / AI 作为工作流一环（location→weather→device→context-fusion→system.ai→rule-engine→music-player）/ AI 推荐人确认后启动。
5. **个性化与隐私**：本地记录 query→理解→人反馈（确认/修改/取消），本地微调，"所有数据留在设备上"；总原则"不上云，不依赖外部 API，跑在设备上"。
6. **零模型兜底**：关键词规则（"下雨天"→scene=rainy_day 等），自称规则匹配率 60-70%、模型 90%+。
7. **实施三期**：规则兜底 1h → ONNX 模型 1 天 → 个性化 1 天。

## 与既有文档的兼容性

**矛盾点：**

1. **类型体系三重冲突（文档无任何说明）**：① `intelligence` 不在 `atom-registry-schema.json:90` 的 8 值枚举（function/rule/external/agent/mcp-server/data/schema/gateway）内；② 也不在 v2 分类计划新增的三值（sensor/fusion/actuator）内——v2 落地后它是**第十一值**，但 v2 文档与本文档互无引用；③ 与 v2 五类对照，"理解人想做什么"最贴近**决策类**（v2:96-105 决策=情境→规则匹配→动作决策），本文档却把它与 rule-engine 并列前置（"rule-engine 做匹配，AI 做理解"），实质是把决策类劈成两半却未修订 v2 分类。另注：v1 枚举本有 `agent` 值可承载 AI 型原子，文档未考虑复用。
2. **"规则兜底"与 rule-engine 规格不一致**：v2 的 system.rule-engine 输入是 `{situation, context}`（结构化情境），规则存于 atom_registry 中 type=rule 的原子；本文档 §七的兜底规则直接吃原始 query 字符串做 `in` 匹配，是**第三种未定义的规则机制**（非 rule-engine 规格、非注册规则原子），文中"规则引擎兜底"的措辞会让读者误以为复用 rule-engine。
3. **"不上云"原则与同仓库 M5 改道并存且互不引用**：本文档头部即宣言"不上云，不依赖外部 API"；而 `embeddings.py:63-102` OpenAIEmbeddingProvider 经 `EMBEDDING_API_BASE/EMBEDDING_API_KEY/EMBEDDING_MODEL` 环境变量把原子功能文本发往 OpenAI 兼容远端（00 总览 P2-10 已记"M5 改道 + 远端 embedding 与 M6 威胁模型有未讨论冲突"）。两者可辩护为分层不同（M5=注册表语义搜索的基础设施层，system.ai=用户 query 的设备侧理解层），但**没有任何文档写下这条分层界线**——按字面原则，M5 远端路径已违反"不上云"。
4. **方案 C 自我违反原则**：Claude CLI 本质是云端 API 调用，与"不上云，不依赖外部 API"直接冲突；文档只指出其 Node.js 依赖问题，对原则冲突只字未提。
5. **"首启下载模型"与原则的张力**：~50MB 模型"首次启动时下载"需要网络出口，与"不依赖外部 API"有程度性张力（一次性下载 vs 持续调用），文档未划界；M6 威胁模型（DESIGN_M6_SECURITY.md §2）覆盖认证/SSRF/文件读取，完全不涉及模型下载的供应链校验（哈希/签名），模型入 APK 亦无校验设计。
6. **依赖链全部悬空且未声明前置**：示例链路的 rule-engine / music-player / weather / location / notification / context-fusion 全部来自未实现的感知层设计（07 报告已确认 12 原子零实现）；映射表中唯一现存目标是 system.file-write（13 基础原子之一）与 search（M5 已实现）。这是 10 报告"示例原子全部来自未实现的感知层"问题的第三次复现，三份文档互挂空链。
7. **与 M5 语义搜索存在未讨论的职责重叠**：`matched_atoms` 输出与 `embeddings.py:236` search_functions（自然语言 query → 函数/原子匹配，api.py:231-249 `/search` 端点已上线）是同一能力的两种实现路线（分类器 vs embedding 检索）；"recommend → search + 工作流推荐"更直接点名复用 search。文档对 M5 只字未提，未论证为何不用既有 embedding 检索而另起分类器。

**未发现矛盾的部分：**

- 命名空间兼容：`system.ai` 走保留前缀内置路线，不触碰注册禁令；全仓 grep `system.ai` 仅本文档命中，无 ID 冲突。
- 编排位置自洽：作为 DAG 前置环节或工作流一环（方式 1/2/3），不违反连线规则的静态依赖形态（它不引入事件推模式，比感知层 12 原子保守）。
- 方案 B（纯关键词）无任何外部依赖，与"不上云"完全自洽，是全文唯一零冲突的可执行部分。

## 与代码现状的差距（带证据）

- **方案 A 的运行时地基不存在（最硬伤）**：`widgetmcp_src/app/build.gradle:23-31` pip 清单仅 fastapi/uvicorn/pydantic/requests/jsonschema/pyyaml/cryptography，无 onnxruntime；DESIGN_CHAQUOPY_MIGRATION.md §五依赖检查也未覆盖它。外部实证：onnxruntime 含 C 扩展且 **Chaquopy 官方包仓库无其 Android wheel**（chaquo/chaquopy issue #216 自 2020 年开放至今，2025-04 的 #1363 仍确认"onnxruntime is in a similar situation"）——`pip install onnxruntime` 在 Chaquopy 下直接失败。可行替代是 Java 侧 onnxruntime-android AAR，但那就不是"Chaquopy 内嵌 Python → ONNX Runtime"的架构，PythonBridge.kt（40 行，仅 start_server）没有任何模型推理通道，文档未评估这条路线分歧。
- **依赖原子零实现**：base-atoms/ 实测仍 13 目录，无 rule-engine/music-player/weather/location/notification/context-fusion；代码侧 grep 上述原子 ID 零命中（07 报告结论本次复核仍成立）；engine 无 trigger/event 机制，方式 2 的级联链不可自动触发。
- **静态检查与执行分发无挂载点**：`workflow.py` BASE_ATOM_IO 仍 13 条，system.ai 落地即吃 "unknown base atom" warning 并落入注册原子跳过路径（07 已述机制）；engine._execute_node 无 system.ai 分发。
- **模型入 APK 无机制**：assets 打包 ~50MB 模型 vs Chaquopy 文档 §六估算 APK 总量 ~30MB——单模型即超出现有 APK 估算 1.6 倍；abiFilters 已裁 ARM 双架构（build.gradle:34）但 Python 侧无 onnxruntime 可用，ABI 裁剪对模型文件无意义；文档无模型校验/更新/回滚设计。

## 风险与建议（按优先级）

1. 【高】技术路线重选：放弃"Chaquopy 内嵌 Python → onnxruntime-python"，改为 Java 侧 onnxruntime-android AAR + Python 仅做前/后处理，或改用纯 Python 可运行的方案（如 ONNX 之外的 sklearn 级小模型）；选型前先写 10 行 spike 验证 pip 可装性。

3. 【高】划清"不上云"分层界线：写一句明确原则（如"用户 query 不出设备；注册表元数据可经配置的远端 provider 处理"），回填到本文档头部、M5 文档与 M6 威胁模型，并给 M6 补数据出境一节。
