## DESIGN_M4_REGISTRY_API.md

**一句话定位**：把注册中心从 Python 函数库升级为 FastAPI HTTP 服务（:8081，与 McpServer :8080 共存，共享 agent.db WAL）。

**核心决策清单**
- FastAPI + uvicorn；端口 8081；共享 SQLite（WAL 避免读写冲突）
- M4 阶段不做认证（推迟到 M6）；响应 < 50ms
- M4.2 前置：`atom_versions` 表 + semver bump 规则（patch/minor/major）
- 模块结构：`api.py` 入口 + `api/` 包（deps/schemas/routes）+ 迁移 `003_atom_versions.py` + 改 `start_yuanzi_termux.sh`

**关键契约**：`GET /api/v1/atoms`（status/category/search/maturity + page/size/sort/order 分页）、`/atoms/{id}` 全字段详情、`/stats`、`/health`（含 db_connected/atom_count/schema_version/uptime）、`/atoms/{id}/versions`。

**文档与代码现状差距**
- 路由**无 `/api/v1` 前缀**，全部挂在根路径（`mcp-yuanzi-bridge/api.py:112-126` `/health` `/stats` `/atoms`）。
- **分页/排序未实现**：`atoms()` 仅支持 status/category/search，无 page/size/sort/order，返回裸 List 而非 `{total,page,items}` 信封（`api.py:120-126`）。
- `/health` 只返回 `{"status":"ok"}`，缺 db_connected/atom_count/schema_version/uptime（`api.py:112-114`）。
- 模块结构不符：无 `api/` 包，单文件 `api.py`（465 行）内联全部 ~30 个路由；迁移是 SQL 而非 Python（`migrations/003_atom_versions.sql` ✅ 存在）；`start_yuanzi_termux.sh` 不存在。
- 超出文档已实现：版本详情/回滚/依赖解析路由（`api.py:203-224`）、`start_server()` Chaquopy 入口端口 8081 与文档一致（`api.py:448`）。
- 完成度：核心查询 API 已实现且有 `tests/test_api.py`、`test_versions.py`；契约形状与文档漂移大。

## DESIGN_M5_SEMANTIC_SEARCH.md

**一句话定位**：自然语言找原子能力——MiniLM 向量 + BM25 融合排序，模型不可用时优雅降级 BM25。

**核心决策清单**
- 本地模型 all-MiniLM-L6-v2（384 维，CPU < 10ms/条）；向量存 SQLite TEXT(JSON)
- 融合分 = 0.7×cosine + 0.3×BM25；模型缺失自动退化 keyword 模式
- `atom_embeddings` 表（迁移 004）；search_text = name+description+purpose+functions+tags
- F4 组合推荐："经常一起出现"的能力推荐

**关键契约**：`POST /api/v1/search`（query/top_k/category/maturity/use_semantic → search_mode/took_ms/matched_functions）、`POST /search/rebuild`、`GET /search/status`。

**文档与代码现状差距**
- **技术路线整体改道**：代码是 Provider 抽象——`MockEmbeddingProvider`（hash-bow-v1，128 维，无真实语义）/ `OpenAIEmbeddingProvider`（远端 OpenAI 兼容接口），**无 MiniLM、无 BM25、无 0.7/0.3 融合排序**（`mcp-yuanzi-bridge/embeddings.py:37-80`）。
- 表为 `function_embeddings`（**函数级**粒度）而非文档的 `atom_embeddings` 原子级（`migrations/004_function_embeddings.sql`）。
- 无 `/search/rebuild`、无 `/search/status`；仅 `GET/POST /search?q=&limit=&provider=mock`，响应无 search_mode/took_ms 字段（`api.py:247-294`）；批量建索引是独立脚本 `embed_atoms.py`。
- F4 实现为依赖/同类别加权推荐（`recommend.py`；路由 `/atoms/{id}/recommendations`、`/combination`，`api.py:226-245`），非文档的共现推荐。
- 完成度：检索链路可跑（`tests/test_search.py`/`test_embeddings.py`），但与文档契约（模型、表、端点、降级语义）几乎不重合，文档需按实现重写。

## DESIGN_M6_SECURITY.md

**一句话定位**：修 Audit 发现的 P0/P1 安全洞（BUG-007/008/016/020/025）——Bearer 认证 + RBAC + probe/SSRF 加固 + content_hash 落库 + 审计哈希链。

**核心决策清单**
- 静态 API Key（不引 OAuth/JWT）；token 来源：env `YUANZI_API_TOKEN` > api_tokens 表 > 皆空则开发模式放行
- 四级角色 admin/registry/viewer/probe；14 个路由权限映射
- probe 仅 http/https + CIDR 白名单（默认 127.0.0.0/8，`YUANZI_PROBE_ALLOWED_CIDR` 可配）
- content_hash/identity_hash 物理列 + 注册查重（duplicate_signature）；审计行 chain_hash 防篡改
- atom-file-read 路径沙箱 / atom-http-get SSRF 防护（100KB/10s）

**关键契约**：`Authorization: Bearer`；`/tokens` CRUD（admin）；`GET /audit/verify`；迁移 005/006（文档 §4 编号）。

**文档与代码现状差距**
- 主体已全部落地：`auth.py` RBAC 含角色等级（`auth.py:100-118`）、`/tokens` CRUD（`api.py:296-319`）、probe scheme+CIDR 白名单（`registry.py:621-687`）、content_hash 物理列+查重+回填（`registry.py:439-462,1172-1186`）、`/audit/verify`（`api.py:433-436`）、base-atom 沙箱（`base-atoms/file-write/core.py:6-39`、`http-get/core.py:26-74`，含 ATOM_FILE_ROOTS 白名单与私网 IP 拒绝）。
- **文档自身编号打架**：§4 说审计链是迁移 006，§6 子任务表说是 007；代码实际 005=content_hash、006=api_tokens、008=audit_chain（`migrations/`）。
- 威胁表"14 个路由"已过时：`api.py` 现约 30 个路由（含 workflow/federation/marketplace）。
- 开发模式（无 token 全放行）仍是默认，仓库内启动脚本均不设置 `YUANZI_API_TOKEN`，安全性依赖部署方自觉。
- 测试现状：`tests/test_auth.py`/`test_audit_chain.py` 等存在；全新 DB 下 262 全过（实测）。但**仓库自带的 `registry.db` 已损坏**：列已有 content_hash 而 schema_migrations 只记到 004，默认库 `import api` 即在迁移 005 崩溃（实测复现），连带 test_api/test_auth 收集失败。

## DESIGN_M7_MARKETPLACE_WORKFLOW.md

**一句话定位**：前 6 阶段总装——原子市场（评分/评论/榜单）+ 工作流 DAG 画布与执行引擎 + 联邦注册中心。

**核心决策清单**
- 综合分 = 0.3×可用性 + 0.2×文档 + 0.4×社区 + 0.1×测试；市场三 Tab（热门/高分/最新）
- 工作流 = 原子+通道的 DAG；保存时 5 条验证（类型匹配/无孤立/无环/必填覆盖/无危险链）
- 执行引擎：拓扑分层执行；状态机 PENDING→RUNNING→SUCCESS/FAILED/TIMEOUT/CANCELLED；节点重试≤3
- 联邦：peer 信任三级（信任/审核/未知），共享元数据不含 endpoint，评分双向同步

**关键契约**：workflow JSON 存储格式（nodes/channels，param 节点）；运行日志 run_id/node_runs。

**文档与代码现状差距**
- 后端基本完成且对齐文档：`marketplace.py:15-18` 权重与文档一致；`workflow.py` 实现全部 5 条验证（`workflow.py:121-231`，危险链为警告级）；`engine.py` 运行实例；`federation.py` 三级信任 + sync（trusted 直接过审，review 进 submitted）；迁移 009/010/011 ✅；REST 路由齐（`api.py:321-431`）；`tests/test_marketplace.py`/`test_workflow.py`/`test_federation.py`。
- **Android 侧零落地**：`widgetmcp_src` 全部 .kt 中无 marketplace/workflow 任何引用（grep 实证），画布编辑器（7.2b，预估 5d）与市场界面（7.1c）未开始——与 PROJECT_PLAN.md"M7 进行中（后端完成，Android 画布待做）"一致。
- 联邦"评分双向同步"（§5.2）未实现：`federation.py` 只同步原子元数据，无 rating/review 传输。
- 重试策略（§4.3 最多 3 次间隔递增）在 `engine.py` 中未见对应机制（文档承诺，代码从简）。
- 完成度：后端约 80%，端到端（含 Android 画布）约 40%。

## DESIGN_M8_HUMAN_EXPERIENCE.md

**一句话定位**：体验层总纲——地基 Graph SDK 零审美、只暴露钩子，模板（Obsidian 星系首发）注入全部审美。

**核心决策清单**

**关键契约**：钩子签名清单（`graph.registerTemplate` / `applyTemplate`）。

**文档与代码现状差距**

## 组内跨文档矛盾与不一致

1. **M8 两份文档钩子清单不一致**：HUMAN_EXPERIENCE §二列 13 个钩子含 `onLayoutTick(forces)`，IMPLEMENTATION 的 `GraphTemplate` 接口无此方法；代码跟随后者——力导向注入力无法实现。
2. **M6 文档内部迁移编号冲突**：§4 数据模型"迁移 006=审计链" vs §6 子任务"M6.4a 迁移(007)"；代码两者皆非（006=api_tokens，008=audit_chain）。
3. **M4/M5 文档假设的 URL 空间（`/api/v1/*`）与 M6 权限映射表、M7 端点均基于此前缀**，而代码无此前缀——四份文档契约互相一致、集体偏离实现。
4. **M5 技术选型（本地 MiniLM+BM25）与 M6 威胁模型冲突未声明**：M6 要求一切外联收敛，而实现选了远端 OpenAI embedding provider（数据出域），两份文档均未讨论。
5. **状态字段集体失真**：六份文档均标 `design-ready`；PROJECT_PLAN.md 标 M4/M5"未开始"、M6"已完成"、M7"进行中"——代码实为 M4/M5/M6 后端均已实现且 262 测试通过，文档与计划文档双双落后于代码。
6. **M7 工作流示例 atom_id 用 `system.http-get` 命名**，实际 base-atoms 目录为 `http-get` 等裸名（`base-atoms/`），命名空间未对齐（RESERVED_PREFIXES 存在于 `registry.py`，需确认 `system.*` 是否为约定前缀）。
