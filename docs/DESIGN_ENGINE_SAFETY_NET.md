# 引擎安全网

> **定位**: 引擎卡住、沉默、输出垃圾时的兜底。不是修原子, 是保引擎不崩溃。

---

## 一、引擎会出什么问题

```
沉默:     共振没结果 → 什么都没有 → 用户不知道发生了什么
混沌:     集群乱组 → 推了完全不对的东西 → 用户困惑
卡死:     工作流卡在某节点 → 停在半路 → 用户不知道在等什么
退化:     权重学到极端值 → 每次输出都一样 → 越来越窄
```

## 二、安全网做什么

```
1. 心跳检测
  引擎 30 秒内应该至少跑一次主循环。
  超时 → 检查哪个环节卡住了。

2. 沉默兜底
  连续 3 次循环无输出 → 不是什么都不说。
  → 用最简单的规则兜底: "这个时间你通常在听歌。要试试吗？"
  → 不聪明, 但不会让人困惑。

3. 混沌检测
  集群成员和上周相比变化太大 (超过 80% 不同) → 可能乱了。
  → 回退到上次稳定配置。
  → 通知: "引擎暂时恢复为上次的设置。新配置需要更多数据。"

4. 卡死恢复
  工作流单节点卡住超过 2 分钟 → 强制 abort 该节点。
  → 跳过, 走 skip 策略。
  → 日志记录, 不阻塞后续。

5. 权重退化检测
  所有权重都超过 0.9 或低于 0.1 → 学了极端值。
  → 重置到初始值 0.5, 重新开始学。
  → 通知: "引擎重新了解你。之前的数据不会丢, 只是重新开始。"

6. 安全模式
  连续检测到 3 种异常 → 引擎进入安全模式。
  → 只用最保守的默认规则。
  → 不共振、不学习、不推送新东西。
  → 通知用户: "引擎进入安全模式。所有功能正常, 但不做新推荐。
     你可以在设置中重置引擎。"
```

## 三、实现

```
每个循环结束, 安全网做一次检查:

  if 主循环超时 > 30s:
      记录 + 重置循环

  if 连续 3 次无输出:
      发送兜底建议

  if 集群变化 > 80%:
      回退 + 通知

  if 工作流卡死 > 2min:
      强制 skip

  if 权重退化:
      重置

  if 异常计数 > 3:
      进入安全模式
```

## 四、修复机制

每种异常有对应的修复动作。

```
沉默修复:
  兜底建议从哪来?
    → 读取 person-atom 的 rhythm 字段
    → 找到当前时段的历史高频行为
    → 生成一句话建议: "这个时间你通常在听歌。要试试吗？"
    → 不共振、不聚类。就是用最简单的方式给用户一个选项。

混沌修复:
  稳定配置存哪?
    → 每 10 次正常循环, 保存一份快照 (集群成员 + 权重)
    → 快照最多保留 5 份
    → 混沌时 → 回退到最近一份正常快照
    → 通知: "引擎暂时恢复。新配置需要更多数据。"

卡死修复:
  怎么强制 skip?
    → 安全网直接调用 workflow_engine.abort_node(node_id)
    → 该节点标记为 skipped, 不阻塞后续
    → 日志记录: "node X aborted by safety_net"

权重退化修复:
  怎么重置?
    → 所有权重 → 0.5 (中性)
    → 保留原始档案数据, 只重置权重
    → 等于重新开始学习, 但不丢历史

安全模式:
  安全模式跑什么?
    → 关闭共振、聚类、学习
    → 只跑默认规则 (时间+基础偏好)
    → 用户可在设置中手动退出安全模式
    → 退出时可选: 保留数据 / 完全重置
```

安全网是独立模块。不和引擎逻辑混在一起。它只是观察引擎, 在需要的时候介入。
