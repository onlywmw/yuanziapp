## 3. 工作流 DAG (7.2)

### 3.1 画布

基于已有的通道模型，工作流是原子 + 线的有向无环图：

```
Android 知识图谱中的工作流画布:

  ┌──────────────────────────────────────┐
  │  工作流: "每日数据采集"              │
  │                                      │
  │  [http-get] ═══[映射]═══ [json-parse]│
  │      │                        │      │
  │      │                   [转换]│      │
  │      │                        │      │
  │  [date-time]────────[合并]──→[file-write]
  │                                      │
  │  [▶ 运行] [💾 保存] [📤 分享]       │
  └──────────────────────────────────────┘
```

### 3.2 节点类型

| 节点 | 图标 | 说明 |
|------|------|------|
| 基础原子 | 灰底圆角方块 | 系统内置 |
| 注册原子 | 白底圆角方块 | 市场安装 |
| 参数节点 | 菱形 | 人工输入的参数 (key, url, pattern) |
| 通道 | 彩色箭头线 | 直通/映射/转换/合并/分流 |
| 条件 | 菱形 | if/else 分支 (未来) |

### 3.3 验证规则

保存工作流时自动验证：

```
1. 连线类型匹配    ← 已定义 (WORKFLOW_CONNECTION_RULES)
2. 无孤立节点       ← 每个原子必须有输入或输出连线
3. 无环             ← DAG 必须是单向无环
4. 必填参数已覆盖   ← 每个原子的必填输入必须有来源
5. 无危险链         ← 已定义 (ATOM_CONNECTION_RULES)
```

### 3.4 存储格式

```
{
  "workflow_id": "wf_daily_collect",
  "name": "每日数据采集",
  "author": "张三",
  "nodes": [
    {"id": "n1", "atom_id": "system.http-get"},
    {"id": "n2", "atom_id": "system.json-parse"},
    {"id": "n3", "atom_id": "system.file-write"},
    {"id": "n4", "atom_id": "system.date-time"},

    {"id": "c1", "type": "map", "source": "n1", "target": "n2", "mapping": {"body": "text"}},
    {"id": "c2", "type": "merge", "source": ["n2", "n4"], "target": "n3"}
  ]
}
```

## 4. 工作流执行引擎 (7.3)

### 4.1 执行模型

```
拓扑排序 → 逐层执行 → 通道转换 → 下一层

  [n1: http-get] ──┐
                    ├─→ [c1: 映射] ──→ [n2: json-parse]
  [n4: date-time] ──┘                       │
                                       [c2: 合并]
                                            │
                                       [n3: file-write] ──→ 完成
```

### 4.2 运行时状态

每个工作流运行实例有状态：

```
PENDING  →  RUNNING  →  SUCCESS
                    →  FAILED (某节点失败)
                    →  TIMEOUT (超时)
                    →  CANCELLED (手动取消)
```

### 4.3 错误处理

```
节点失败策略:
  · 重试: 最多 3 次, 间隔递增
  · 跳过: 可选节点允许失败
  · 终止: 关键节点失败 → 整个工作流标记 FAILED

通道阻塞:
  · 类型转换失败 → 标记通道为红色, 工作流暂停
  · 超时 → 标记通道为黄色, 重试
```

### 4.4 执行日志

```
每个工作流运行产生日志：

{
  "run_id": "run_001",
  "workflow_id": "wf_daily_collect",
  "started_at": "...",
  "status": "SUCCESS",
  "node_runs": [
    {"node": "n1", "status": "SUCCESS", "duration_ms": 230, "output_size": 4096},
    {"node": "n2", "status": "SUCCESS", "duration_ms": 45, "output_size": 512},
    {"node": "n3", "status": "SUCCESS", "duration_ms": 120, "output_size": 0}
  ]
}
```

## 5. 联邦注册中心 (7.4)

### 5.1 概念

多个 Yuanzi 实例可以互相发现和共享原子：

```
注册中心 A (Yuanzi 实例 1)     注册中心 B (Yuanzi 实例 2)
┌─────────────────────┐       ┌─────────────────────┐
│ mcp.postgres (张三)  │ ←──→ │ mcp.mongodb (赵六)   │
│ mcp.mysql (李四)     │       │ mcp.redis (孙七)     │
│ system.* (内置)      │       │ system.* (内置)      │
└─────────────────────┘       └─────────────────────┘
```

### 5.2 共享模型

```
共享原子时传输的信息:
  · 原子元数据 (atom_id, name, author, purpose, signature)
  · 评分和评论 (可选)
  · 不传输运行时数据 (endpoint 是本地概念)

共享协议:
  1. 实例 A 声明自己开放联邦
  2. 实例 B 添加 A 为信任对等节点
  3. 定期同步: A 的新增/更新原子 → B 可以看到
  4. 评分双向同步
```

### 5.3 信任模型

```
信任级别:
  · 信任: 原子自动在对方市场可见, 评分同步
  · 审核: 原子需对方管理员审核后才可见
  · 未知: 不共享, 仅单向搜索
```
