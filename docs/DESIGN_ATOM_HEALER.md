# 原子自愈系统

> **定位**: 原子运行时的"免疫系统" — 检测、修复、隔离、通知
> **区别**: Fixer 修 CI/构建 · Healer 修运行时 · 两者互补

---

## 一、故障分级

```
L0 瞬断    连不上, 但重试就恢复         自动重试 3 次
L1 降级    部分功能坏了, 核心功能还在     标记 degraded, 继续用
L2 故障    核心功能坏了, 原子不可用       自动修复 (重启/回滚)
L3 损坏    数据坏了, 或行为异常           隔离 + 通知作者
L4 感染    恶意代码, 或攻击行为           立即隔离 + 社区告警
```

## 二、修复策略

### L0-L1: 自动处理, 人不感知

```
probe 检测到 L0:
  → 重试 3 次 (间隔 1s, 3s, 10s)
  → 恢复 → 记录 "瞬断 1 次", 不告警
  → 3 次都失败 → 升级 L2

probe 检测到 L1:
  → 标记 status = "degraded"
  → 继续使用 (核心功能仍可用)
  → 通知作者 "部分功能异常"
```

### L2: 自动修复

```
按修复链依次尝试:

  1. 重启原子 (重新调用 handler)
  2. 回滚到上一个正常版本 (rollback_atom)
  3. 重装依赖 (如果声明了 dependencies)
  4. 替换为同功能原子 (content_hash 相同但 atom_id 不同)

每一步后 probe 验证。
修复成功 → 记录 "自愈成功" + 恢复正常状态。
全部失败 → 升级 L3。
```

### L3: 隔离

```
隔离一个损坏的原子:

  1. status → "quarantined"
  2. 断开所有连线 (不再接收/发送数据)
  3. 图谱上显示为红色 ×, 不可交互
  4. 通知作者 (邮件/站内信)
  5. 通知依赖此原子的其他原子 → 降级或暂停

隔离不是删除。数据保留, 等作者修复。
```

### L4: 紧急隔离

```
检测到恶意行为 (异常调用频率、非法数据访问):

  1. 立即隔离 (不等 probe 周期)
  2. 撤销该原子的所有权限
  3. 通知社区 (该原子被标记为"安全风险")
  4. 下游原子自动回退到上一版本或替换
  5. 证据保全 (审计日志)
```

## 四、图谱上的表现

```
正常运行:      节点正常色, 连线正常
L0 重试中:     节点微闪 amber (重试动画)
L1 降级:       节点 amber, 边框虚线
L2 修复中:     节点 amber + 脉动动画, 连线断开
L3 隔离:       节点红色 ×, 灰色底色, 无连线
L4 紧急隔离:   节点红色 × + 闪烁, 所有连线断裂动画

修复成功:      节点绿色闪一下 → 恢复原色, 连线重连动画
修复失败→L3:   节点红色蔓延 → 红色×
```

## 五、通知链路

```
L0: 不通知
L1: 静默通知作者 (站内信, 不打扰用户)
L2: 通知作者 + 依赖此原子的用户 "正在自动修复"
    修复成功 → "已恢复"
    修复失败 → "已隔离, 请联系作者"
L3: 通知作者 + 所有依赖者 + Audit
L4: 通知作者 + 所有用户 + Audit + 社区公告
```

## 六、预防

```
自愈系统的数据反过来强化预防:

  统计:
    最常出现的故障类型 → 优化注册时的验证规则
    最常故障的原子 → 降低其评分
    最常故障的依赖组合 → 工作流连线时警告

  注册前检查:
    原子是否引用了已知不稳定的依赖？
    原子的 endpoint 是否在过去 7 天内稳定？
    作者的历史原子是否有较高的故障率？
```

```json
      "status": "healthy",

    }
  }

```

审计表新增 `incident_id` 关联字段。

## 八、实施

```
P0 (必须有):
  · L0 自动重试
  · L2 重启 + 回滚
  · L3 隔离通知
  · 图谱故障动画

P1 (应该有):
  · L2 依赖替换
  · L4 紧急隔离
  · 自愈日志面板

P2 (锦上添花):
  · 预防分析
  · 社区告警
  · 故障预测
```
