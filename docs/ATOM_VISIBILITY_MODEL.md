# 原子可见性模型

> **核心**: 不是所有原子都需要在首页展示。存在即可, 展示是特权。
> **原则**: 人在图谱上看到的应该是"对自己有意义的原子", 不是系统的全部内脏。

---

## 一、三层可见性

```
┌─────────────────────────────────────────────┐
│  Layer 1: 展示层 (默认可见)                  │
│  终端原子 — 商品/服务/作品                    │
│  人浏览、发现、收藏的原子                     │
│  ≈ 图谱默认视图                              │
├─────────────────────────────────────────────┤
│  Layer 2: 折叠层 (搜索可见)                  │
│  基础设施原子 — 数据库/云服务/系统工具         │
│  存在即可, 只在搜索或作为依赖时出现            │
│  ≈ 图谱"显示全部"模式                         │
├─────────────────────────────────────────────┤
│  Layer 3: 隐藏层 (不可见图谱)                │
│  基础原子 — system.*                         │
│  系统内置, 永远不可见图谱, 只在依赖列表中显示   │
│  ≈ API 可查, 图谱不绘                        │
└─────────────────────────────────────────────┘
```

---

## 二、分类映射

```
原子类别            默认可见    搜索可见    图谱渲染
─────────────────────────────────────────────────
system.* (基础原子)    ❌         ❌         不渲染
mcp.* (数据库类)       ❌         ✅         折叠层
mcp.* (云服务类)       ❌         ✅         折叠层
mcp.* (安全类)         ❌         ✅         折叠层
终端原子               ✅         ✅         展示层
作者精选               ✅ (大)     ✅         展示层高亮
```

### 数据库/云服务/安全 = 基础设施, 不是终端产品

```
这些不需要展示在首页:
  mcp.postgres     ← 数据库连接器
  mcp.mysql        ← 数据库连接器
  mcp.s3           ← 云存储
  mcp.iam          ← 权限管理
  mcp.cloudwatch   ← 监控
  mcp.security-agent ← 安全工具

这些是"管道", 不是"产品"。
人不需要在首页看到管道。
```

## 三、Schema 扩展

在 `classification` 中新增 `visibility` 字段：

```json
{
  "classification": {
    "category": "Database",
    "visibility": "infrastructure",    ← 新增
    "soul": { ... }
  }
}
```

```
visibility 取值:
  "terminal"       终端原子 — 默认展示在首页 (商品/服务/作品)
  "infrastructure" 基础设施 — 存在即可, 搜索时才出现
  "system"         系统原子 — 不可见图谱 (基础原子用)
  "featured"       精选 — 首页放大展示 (由社区或管理员标记)
```

### 默认值

```
已有 61 原子:
  visibility = "infrastructure" (保守: 全部折叠, 不破坏现有体验)

新原子:
  作者注册时选择:
    · "这是一个终端产品/服务/作品" → terminal
    · "这是一个基础设施/管道工具" → infrastructure

基础原子:
  visibility = "system" (不可改)
```

---

## 四、图谱渲染规则

```
默认视图:
  只渲染 visibility = "terminal" 或 "featured"

"显示全部" 模式:
  渲染 terminal + infrastructure
  system 仍然不渲染

搜索时:
  无视 visibility, 所有匹配结果都显示
  搜索到的 infrastructure 原子临时出现在图谱上
  搜索结果关闭后恢复隐藏

局部图谱:
  双击 terminal 原子 → 显示其邻居
  即使邻居是 infrastructure → 也显示 (因为是依赖关系)
```

---

## 五、API 变更

```
GET /atoms
  默认: visibility = "terminal" | "featured"  (不需要参数)
  ?visibility=all → terminal + infrastructure
  ?visibility=system → 所有 (管理接口, admin only)

GET /atoms/{id}
  无变更 — 单个原子详情始终可查, 无论 visibility
```

## 七、终端原子的判定

```
一个注册原子被认为是"终端"的条件 (满足任一):

  1. 填写了 soul 字段 (有风格/受众/叙事)
  2. 作者标记为 "terminal"
  3. 社区评分 ≥ 4.0 且评价 ≥ 10
  4. 被管理员标记为 "featured"

默认: 新注册原子 → infrastructure
升级: 作者填写 soul 或获社区认可 → 自动变 terminal
```

## 八、实施

```
改动点:
  atom-registry-schema.json  + visibility 字段      5 min
  registry.py                注册时读 visibility     10 min
  GraphEngine                按 visibility 过滤节点   15 min
  api.py                     GET /atoms 支持 ?visibility 10 min

零破坏:
  现有原子默认 infrastructure → 和当前行为一致 (全部可见)
  基础原子 system → 本来就不可删除, 现在也不可见
  新原子可选择 terminal → 首页只展示它们
```

---

> **图谱不是仓库清单。首页是橱窗。基础设施在后台安静运行, 终端产品在橱窗里发光。**
