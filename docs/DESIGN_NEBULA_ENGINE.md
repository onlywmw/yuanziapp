# 星云引擎 — 运转设计

> **定位**: 引擎是活的。不停循环。观察 → 共振 → 集群 → 输出 → 学习 → 再观察。
> **区别**: 前面几份说的是原子怎么定义。这份说的是引擎怎么跑。

---

## 一、主循环

```
引擎不休眠。每 30 秒跑一次主循环:

  ┌─────────────────────────────────────────┐
  │                                         │
  │  1. 采集 ── 所有原子, 此刻的场是什么     │
  │      │                                   │
  │  2. 共振 ── 谁和谁在共振, 强度多少       │
  │      │                                   │
  │  3. 聚类 ── 共振最强的形成临时集群        │
  │      │                                   │
  │  4. 决策 ── 这个集群要不要变成输出        │
  │      │                                   │
  │  5. 输出 ── 推给人, 或默默作为上下文      │
  │      │                                   │
  │  6. 反馈 ── 人接受了、忽略了、还是调整了  │
  │      │                                   │
  │  7. 学习 ── 更新权重, 回到 1              │
  │                                         │
  └─────────────────────────────────────────┘
```

## 二、30 秒一次, 不是实时

```
不需要毫秒级。人不是高速机器。

30 秒一次:
  0-5ms:   采集 (读 12 个原子的场)
  5-7ms:   共振计算 (66 对 × 5 维度)
  7-8ms:   聚类
  8-9ms:   决策
  剩下 29.99 秒: 等待下一个循环

大部分时间引擎是安静的。
只在需要的时候才动。
```

## 三、采集 — 所有原子同时发声

```
不是"触发哪个工作流"。
是每个循环, 所有原子同时发出自己的场。

  此刻:
    weather:  {雨强:0.8, 密度:0.9, 能量:0.3}
    music:    {慢:0.7, 爵士:0.9, 能量:0.4}
    person:   {需要安静:0.9, 冷:0.8, 不赶:0.7}
    calendar: {空闲:1.0, 持续:2h}
    device:   {降噪:0.9}
    ...

原子不需要被"调用"。
引擎采集它们。它们总是醒着的。
```

## 四、共振 — 谁和谁在一起有意义

```
上一轮: person 和 coffee 共振最强 (冷, 需要暖)
这一轮: person 和 music  共振变强 (暖了, 需要声音)
下一轮: person 和 calendar 共振最强 (准备走了)

每 30 秒, 共振图都在变化。
因为人的状态在变化。
因为原子的状态在变化。

引擎看到的不是"什么触发了什么"。
看到的是一张每 30 秒更新一次的共振热力图。
```

## 五、聚类 — 自然形成的临时组合

```
共振超过阈值的原子自然聚在一起:

  集群 A:  person + coffee + weather         (回应: 暖, 安静)
  集群 B:  person + music + calendar          (回应: 慢, 不赶)
  集群 C:  device + person                    (回应: 不被干扰)

三个集群同时存在。不冲突。各做各的。

集群 A 的输出: 不需要输出。coffee 已经是暖的。它只是被"注意到"了。
集群 B 的输出: 一个 45 分钟的慢爵士列表。刚好够你暖起来。
集群 C 的输出: 不需要输出。降噪已经开着。系统只是知道你在专注。

不是每个集群都产生输出。
有些集群就是"此刻的状态"本身。
```

## 六、输出 — 不是每次都要说话

```
引擎不总是输出。

  以下情况不说话:
    · 集群只是"确认了人的当前状态" — 不需要提醒
    · 集群和上一轮基本一样——没变化就不打扰
    · 人在专注模式——不发 push
    · 深夜——只做后台, 不亮屏

  以下情况轻轻说:
    · 出现了一个上一轮没有的集群 → 人的状态变了
    · gap 在缩小 → "今天比昨天早了"
    · 一个被收藏但没听的列表, 现在时机对了
```

## 七、学习 — 不是训练, 是痕迹

```
每次循环结束:

  如果人接受了集群的输出:
    参与共振的维度权重 +0.05
    集群的成员组合被记录为"有效模式"

  如果人忽略了:
    权重 -0.03
    模式被标记为"这次不适用"

  如果人什么都没做:
    不调整权重
    这是沉默——也是信号

不是机器学习。不需要 optimize()。
就是每次循环后, 轻轻地记一笔。
```

## 八、引擎状态

```
引擎有状态。不是无状态的函数调用。

  idle:     正常循环, 每 30 秒一次
  noticing: 发现了一个变化——稍微提高采集频率 (5 秒一次)
  suggesting: 准备输出 → 轻轻告诉人
  waiting:  等人的反馈 (30 秒窗口)
  learning: 收到反馈 → 更新权重
  resting:  深夜/人忙/不打扰 → 降到 5 分钟一次

状态切换图:
  idle → noticing → suggesting → waiting → learning → idle
    ↑                                              |
    └──────────────────────────────────────────────┘
```

## 九、引擎和原子的关系

```
不是引擎"调度"原子。

引擎像一个观察者, 站在所有原子中间。
每 30 秒看一眼: "你们现在什么状态？谁和谁在一起了？"

引擎不创造连接。它只是看到连接。
引擎不决定输出。它只是传递输出。
引擎不学习。它只是记录痕迹。

原子自己共振、自己聚合、自己发出输出。
引擎只是那个让它们有机会相遇的地方。
```
