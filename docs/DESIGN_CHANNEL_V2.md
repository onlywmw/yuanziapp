# 通道体系升级

> **定位**: 通道从"连线的属性"升级为"一等原子"——可注册、可复用、可测试、可发现
> **基础**: DESIGN_CHANNEL_SPEC.md 的 5 种通道 + 5 个硬伤修复
> **升级**: 通道可注册、可复用、智能推荐、运行时可视化、通道市场

---

## 一、通道即原子

```
旧模型:
  通道 = 工作流连线的属性, 绑在 workflow JSON 里, 没法单独存在

新模型:
  通道 = system.channel.* 原子, 和 math-calc、http-get 一样
  可以单独注册、单独测试、单独搜索、被任何人复用
```

```
system.channel.map      映射通道
system.channel.merge     合并通道
system.channel.split     分流通道
system.channel.transform 转换通道
system.channel.direct    直通通道

使用:
  工作流画布上拖一根线 → 选择通道类型 → 配置 mapping/strategy
  → 本质是插入了一个 system.channel.map 原子

  保存时, 这个通道原子被注册 (如果有复用价值) 或内联 (一次性使用)
```

### 通道原子和普通原子的区别

```
相同:
  · 有 atom_id, 有 I/O, 有版本, 有签名
  · 可以被搜索、被安装、被复用

不同:
  · 没有独立 handler — 它的 handler 是系统内置的映射/合并/转换函数
  · 不是独立执行 — 它在工作流中作为两个原子之间的桥
  · 没有独立的探针 — 它的健康度由所在工作流的成功率决定
```

## 二、通道复用

```
张三创建了一个通道:
  system.channel.map
    source: system.weather
    target: system.context-fusion
    mapping: {condition: $.condition, temp: $.temperature}
    name: "天气→情境映射"

张三点击 [发布为可复用通道]
  → 注册为 registered_channel.zhangsan.weather-to-context
  → 出现在通道市场

李四在画布上连接 weather → context-fusion 时:
  系统提示: "有 3 个可复用通道:
    · 张三的天气→情境 (128 人使用, ⭐4.8)
    · 王五的天气→情境 (89 人使用, ⭐4.5)
    · 自定义"
  点击 → 一键安装 → 通道出现在自己的连线上
```

### 复用通道的版本

```
张三更新了映射规则:
  weather-to-context v1 → v2
    改动: 新增 humidity 映射

  李四的工作流用的是 v1。
    收到通知: "张三的天气→情境有更新 (v1→v2)"
    李四选择:
      [升级到 v2]  自动更新
      [保持 v1]    继续用旧版
      [查看变更]   对比 v1 vs v2 的 mapping 差异
```

## 三、智能通道推荐

```
画布上, 用户从 A 拖一根线到 B:

系统分析:
  · A 的输出 schema: {body: string, status_code: int}
  · B 的输入 schema:  {text: string}

  → 类型匹配: A.body (string) ≈ B.text (string) ✅
  → 建议映射: {text: $.body}
  → 展示: "自动映射: body → text。确认?"

  用户点确认 → 通道自动创建。

  → 如果社区有 80% 的连接用同样的映射 → 直接显示:
    "80% 的人用这个映射。一键应用?"
```

### 推荐引擎的数据来源

```
1. I/O schema 类型匹配 (自动)
2. 字段名相似度 (body ≈ text → 建议映射)
3. 社区使用频率 (80% 的人用这个 → 推荐)
4. 历史成功/失败统计 (这个映射的错误率低 → 推荐)
```

## 五、通道健康度

```
每个可复用通道有健康度评分:

  健康度 = 成功率(50%) + 平均耗时/P99(20%) + 使用量(15%) + 评分(15%)

  成功率:
    通道在工作流中执行时, 统计映射/合并/转换是否成功

  为什么需要:
    张三的通道有 bug → 成功后率下降 → 健康度 < 50 → 其他用户看到警告
    "此通道最近成功率 67%, 建议等待作者修复"

  不是阻止使用, 是让人知道风险。
```

## 六、通道与发现层的关系

```
基础层 (这里):
  通道是什么、怎么定义、怎么执行 — 纯技术规格

发现层 (DESIGN_M7_MARKETPLACE_WORKFLOW.md):
  通道怎么被发现、怎么推荐、怎么安装、怎么评分
  和原子的发现、安装、评分是同一套机制
  通道作为 type=channel 的原子出现在市场中
```

通道本身不定义怎么被发现。那是 M7 的事。
