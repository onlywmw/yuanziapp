# 原子灵魂与可见性

> **定位**: 从"能用"到"对人有用"——原子怎么被人发现、怎么让人共鸣、怎么自然浮现
> **版本**: v2 — 夯实版。不只是字段定义, 是完整的浮现引擎。

---

## 一、灵魂六字段

### 字段总览

| 字段 | 类型 | 上限 | 权重 | 为什么重要 |
|------|------|------|------|-----------|
| `narrative` | 字符串 | 10-200字 | 3.0 | 最强信号 — 人花时间写的, 不是自动生成的 |
| `style` | 数组 | 3个 | 1.5 | 让人找到"对味"的原子 |
| `audience` | 数组 | 3个 | 1.0 | 让人知道"这是为我做的" |
| `mood` | 字符串 | 单选 | 0.5 | 传达使用时的感觉 |
| `quality` | 字符串 | 单选 | 0.5 | 让人知道"能不能放心用" |
| `use_case` | 数组 | 3个 | 0.5 | 让人知道"什么时候用" |

### 每个字段: 可选值 + 好例子 + 差例子 + 校验

---

#### style (风格标签)

```
可选值:  极简 · 可靠 · 专业 · 优雅 · 强大 · 轻量
         创意 · 温馨 · 硬核 · 极客 · 玩趣 · 实验

好:
  [极简] [可靠] [专业]
  → 三个词勾勒出一个气质: "一个稳重的工具"

  [创意] [玩趣] [轻量]
  → "一个有趣的小东西"

差:
  [极简] [可靠] [专业] [优雅] [强大] [轻量]  ← 全选 = 没选
  [极简] 但原子有 15 个配置项                ← 标签骗人

校验:
  · 不在可选值 → 拒绝
  · 超过 3 个 → 拒绝
  · 选了 5+ 个 (绕过上限的变通) → 星级 -0.5, score 不计 style 分
```

#### audience (受众标签)

```
可选值:  后端开发 · 前端开发 · 数据工程 · 设计师
         作家 · 学生 · 所有人 · 极客 · 运维 · 研究员 · 创作者

好:
  [后端开发] [数据工程]
  → 清楚: "这是给后端和数据的人用的"

差:
  [所有人] [学生] [极客] [创作者] [设计师]
  → 谁都是 = 谁都不是

校验:
  · 不在可选值 → 拒绝
  · 超过 3 个 → 拒绝
  · 包含 "所有人" 且选了 3 个 → 星级 -0.5, score 不计 audience 分
    (选了"所有人"又选特定群体 = 矛盾)
```

#### mood (使用基调)

```
可选值:  专注 · 平静 · 精力充沛 · 轻松愉快 · 严肃认真 · 受启发

单选。不填 = 未声明。
```

#### quality (品质等级)

```
可选值:  experimental · functional · polished · battle-tested · handcrafted

单选。不填时默认 = `functional`。

`handcrafted` 需要人工审核 (防止滥用)。
作者选 `handcrafted` → Audit 抽查 → 确认手工品质 → 标签生效。
```

#### use_case (使用场景)

```
可选值:  日常工作 · 生产环境 · 学习 · 原型开发 · 创意项目 · 紧急救火

最多 3 个。

校验:
  · quality = `experimental` 又选 use_case = `production`
    → 冲突警告: "实验品不建议用于生产环境"
```

#### narrative (作者叙事)

```
一句话, 人写的。不是 AI, 不是模板。

好:
  "写了十年 SQL 的人, 把最趁手的工具分享出来"
  "从自己的项目中抽出来的, 用了三年没问题"

差:
  "这是一个数据库工具"                    ← 和 description 重复
  "PostgreSQL database operations"        ← 英文且和 description 一致
  "非常非常好用, 强烈推荐!!!"             ← 空洞, 无信息量
  "test 123 abc"                          ← 明显占位

校验:
  P0 拒绝:
    · < 10 字 或 > 200 字
    · 和 description 文本完全一致 (忽略大小写)

  P1 警告 (星级 -0.5, score 不计 narrative 分):
    · 含 "test" / "测试" / "123" / "abc" / "todo" (不区分大小写)
    · 纯重复字符 (如 "好好好好好")
    · description 和 narrative 的编辑距离 < 5 (高度相似)
```

## 二、浮现引擎

### 不是简单的加分——是动态的、有衰减的

```
旧版: score = 字段有就加分, 一次计算, 永远不变

新版: score = 灵魂完整度 + 时效衰减 + 作者信誉 + 社区反馈

灵魂不会永远浮在上面 — 旧的会下沉, 新维护的会上浮。
```

### 浮现分数公式

```
visibility_score = soul_score × freshness × author_boost + community_score

soul_score (灵魂完整度, 0~7.0):
  narrative 非空且通过 P1          +3.0
  style 非空且通过校验              +1.5
  audience 非空且通过校验           +1.0
  mood 非空                        +0.5
  quality ≥ polished               +0.5
  use_case 非空                    +0.5

freshness (时效衰减, 0.3~1.0):
  最近 30 天内更新过 soul          1.0
  30-90 天未更新                   0.8
  90-180 天                        0.6
  180-365 天                       0.4
  > 1 年                          0.3

  每次作者修改 soul → freshness 重置为 1.0。

author_boost (作者信誉, 1.0~1.5):
  作者有 10+ 个 ★★★★☆ 原子        1.5
  作者有 5+ 个 ★★★★☆ 原子         1.2
  作者有认证                       1.3
  新作者 (无历史)                  1.0

community_score (社区反馈, 0~1.5):
  评分人数 ≥ 50 且均值 ≥ 4.5       1.5
  评分人数 ≥ 10 且均值 ≥ 4.0       1.0
  评分人数 ≥ 10 且均值 < 3.0       -0.5
  近 30 天有使用记录               +0.3
```

### 为什么需要衰减

```
一个 2 年前写的 narrative:
  "这是第一个版本, 还不太完善"

但原子已经迭代了 15 个版本。
soul 没更新 → narrative 已经说谎了 → freshness=0.3 → score 被压低

倒逼作者: soul 也要维护, 不只是代码。
```

### 为什么需要作者信誉

```
张三: 10 个 ★★★★☆ 原子, 社区认可的作者
李四: 新注册, 第一个原子

张三的新原子 → author_boost=1.5 → 初始可见度高
李四的新原子 → author_boost=1.0 → 需要靠 soul 质量和社区反馈慢慢爬

信誉不是永久的 — 如果张三的原子频繁被举报 → author_boost 降低。
```

### 新原子的冷启动

```
新作者, 第一个原子, soul 完整:
  soul_score=7.0, freshness=1.0, author_boost=1.0, community=0
  → 最终 7.0 → 展示层, 首页可见

系统给新原子一个公平的机会: 前 7 天 freshness 强制 = 1.0。
7 天后如果社区无反馈 → freshness 开始衰减。
```

## 三、和混音台的联动

### 不是"× 权重"——是重新分配可见度

```
混音台位置:

  管道 (0.0):
    只显示 infrastructure 类型原子 (mcp.*, 数据库, 云服务)
    终端原子 (有 soul) 不可见
    可见度 = score × 0.0 = 0  (对于终端原子)

  均衡 (0.5):
    都显示, 正常 score

  作品 (1.0):
    只显示有 soul 的终端原子
    管道原子 visible=0

实现:
  不是简单乘法 — 混音台决定了"哪类原子参与可见度竞争"。
  管道端: 只有管道原子参与, 按各自 score 排序。
  作品端: 只有终端原子参与, 按各自 score 排序。
  中间: 所有原子参与, 按 score 排序。
```

## 四、作者指南

### 新手引导

```
注册表单中的顺序不是按字段排列 — 是按思维流程:

  Step 1: "这个原子是给谁用的?"        → audience
  Step 2: "用起来什么感觉?"             → style + mood
  Step 3: "做到什么程度了?"             → quality
  Step 4: "什么时候用?"                 → use_case
  Step 5: "用一句话介绍它 (你手写)"     → narrative

每一步有 1-2 句引导文字 + 示例。
不是填表格, 是回答自己的问题。
```

### 质量提示

```
填写时实时反馈:

  narrative 写了 8 个字 → "还差 2 个字, 多说一点?"
  style 选了 4 个       → "最多 3 个, 去掉哪一个?"
  quality=experimental + use_case=production → "实验品用于生产——你确定吗?"

不阻断, 只提示。
```

## 五、校验规则汇总

```
P0 拒绝 (注册不通过):
  style/audience/use_case > 3 个
  narrative < 10 或 > 200 字
  narrative 和 description 完全相同
  mood/quality 不在可选值
  quality=handcrafted 且未经 Audit 审核

P1 处罚 (星级 -0.5, 相应字段 score 不计):
  narrative 含占位词 (test/测试/123/todo)
  narrative 纯重复字符
  narrative 和 description 编辑距离 < 5
  style 选 > 5 个 (可绕过上限的场景)
  audience 含 "所有人" 且选了 3 个
  quality=experimental 又选 use_case=production

P2 不干预:
  soul 完全为空 (score=0, 作者选择不填)
```
