# 原子灵魂与可见性

> **定位**: 从"能用"到"对人有用"——原子怎么被人发现、怎么让人共鸣、怎么自然浮现
> **合并**: ATOM_SOUL_MODEL + SOUL_MODEL_IMPLEMENTATION + ATOM_VISIBILITY_MODEL

---

## 一、灵魂字段

### 为什么需要

```
技术视角 (冷):                    人的视角 (有灵魂后):
─────────────                    ─────────────────────
mcp.postgres                     "一个稳如老狗的数据库工具"
7 个功能                          "适合后端开发, 生产级品质"
分类: Database                    "简洁、可靠、不花哨"
作者: 张三                         "来自一个写了十年 SQL 的人"

功能描述让人找到 "能用" 的原子。
灵魂描述让人找到 "属于自己的" 原子。
```

### 六字段

| 字段 | 类型 | 说明 | 必填 | 上限 |
|------|------|------|------|------|
| `style` | 数组 | 风格标签 | 否 | 3 个 |
| `audience` | 数组 | 目标受众 | 否 | 3 个 |
| `mood` | 字符串 | 使用基调 | 否 | 单选 |
| `quality` | 字符串 | 品质等级 | 否 | 单选 |
| `use_case` | 数组 | 使用场景 | 否 | 3 个 |
| `narrative` | 字符串 | 作者手写叙事 | 否 | 10-200 字 |

### 风格标签

```
一个人面对两个功能一样的原子, 怎么选?

  原子 A: minimalist, clean, fast
  原子 B: feature-rich, configurable, powerful

风格让人找到"对味"的那个。

可选值:
  极简 · 可靠 · 专业 · 优雅 · 强大 · 轻量
  创意 · 温馨 · 硬核 · 极客 · 玩趣

最多选 3 个。
```

### 怎样算好的风格标签

```
好: 选 2-3 个, 每个都准确描述这个原子的气质
    [极简] [可靠] [专业]
    → 人一眼就知道这个原子的气质

差: 全选, 没有取舍
    [极简] [可靠] [专业] [优雅] [强大] [轻量] [创意] ...
    → 等于没选, 没有区分度

差: 标签不匹配实际
    [极简] 但原子有 15 个配置项
    → 人用了会觉得上当

校验规则:
  · 最多 3 个, 超过拒绝
  · 标签不在可选值列表中 → 拒绝
  · 全选超过 5 个 → 不拒绝, 但星级 -0.5
```

### 受众标签

```
这个原子是为谁做的?

可选值:
  后端开发者 · 前端开发者 · 数据工程师
  设计师 · 作家 · 学生 · 所有人 · 极客
  运维 · 研究员 · 创作者

最多选 3 个。
全部选"所有人" → 等于没说, 星级 -0.5。
```

### 基调

```
用这个原子的时候, 人是什么感觉?

可选值:  专注 · 平静 · 精力充沛 · 轻松愉快 · 严肃认真 · 受启发

单选。不填 = 未声明。
```

### 品质

```
这个原子做到什么程度了?

可选值:
  `experimental`   实验性的, 可能随时改
  `functional`     能用的
  `polished`       打磨过的, 细节到位
  `battle-tested`  久经考验
  `handcrafted`    匠心手作 (作者亲自标注)

单选。不填 = `functional` (默认)。
```

### 使用场景

```
什么情况下用这个原子?

可选值:
  日常工作 · 生产环境 · 学习 · 原型开发 · 创意项目 · 紧急救火

最多选 3 个。
```

### 叙事

```
一句话, 人写的。不是 AI 生成的。

好的叙事:
  "写了十年 SQL 的人, 把最趁手的工具分享出来"
  "从自己的项目中抽出来的, 用了三年没问题"
  "为女儿做的第一个小程序"

差的叙事:
  "这是一个数据库工具"           ← 没用, 功能描述已经说过了
  "PostgreSQL database operations" ← 没用, 英文且和 description 重复
  "非常非常好用, 强烈推荐"        ← 空洞
  "test test 123"                 ← 垃圾

校验:
  · 10-200 字, 超出拒绝
  · 不能和 description 完全一致
  · 不能包含 "test" / "测试" / "123" 等明显占位词
  · 纯英文且 description 也是英文 → 警告, 不拒绝
```

## 二、可见性: 自然浮现

### 原则

```
不打标签, 不人工分类。

soul 饱满 ──→ 自然浮到上层, 节点更大
soul 半满 ──→ 中层, 正常大小
soul 空白 ──→ 底层, 节点缩小
system.*  ──→ 不可见, 不可操作

像水里的气泡 — 饱满的上浮, 空心的下沉。
```

### 浮现分数

```
每位原子有一个 visibility_score, 纯数据驱动:

  narrative 非空              +3.0    (最有分量的信号)
  style 非空                  +1.5    (作者认真选了风格)
  audience 非空               +1.0    (作者知道为谁做的)
  mood 非空                   +0.5
  quality ≥ polished          +0.5    (打磨过以上的品质)
  use_case 非空               +0.5
  社区评分 ≥ 10 人             +1.0    (社区认可)
  评分均值 ≥ 4.0               +0.5
  system.*                    分数 = 0  (强制)

总分: 0 ~ 8.5
```

### 分数 → 渲染

```
score ≥ 4.0:     展示层 — 首页默认可见, 完整节点
score 2.0 - 3.9: 折叠层 — "显示全部"后可见, 缩小节点
score 0.1 - 1.9: 隐藏层 — 搜索或作为依赖时才出现
score = 0:       不可见 — system.*, 永不绘制
```

### 和混音台的联动

```
混音台位置 (管道 ◄──●──► 作品) 影响可见度权重:

  管道端 (0.0):  管道权重 = 1.0, 作品权重 = 0.0
    → 基础设施原子可见, 终端原子不可见

  中间 (0.5):    管道权重 = 0.5, 作品权重 = 0.5
    → 两者都可见

  作品端 (1.0):  管道权重 = 0.0, 作品权重 = 1.0
    → 终端原子可见, 基础设施原子不可见

最终可见度 = score × 混音台权重 × 力导向节点大小系数
```

### 一个原子的成长

```
刚注册 (soul 为空):
  score = 0 → 隐藏层, 只有搜索能找到

作者填写 soul:
  score = 1.5 → 隐藏层顶部

soul 完整 + 社区开始使用:
  score = 3.5 → 折叠层, "显示全部"可见

叙事丰满 + 高分好评:
  score = 6.5 → 展示层, 首页可见, 节点正常大小
```

## 三、作者指南

### 新手填写顺序

```
1. 先写 narrative (叙事)
   → 回答: "我为什么做这个原子?"
   → 这是灵魂里最重要的字段

2. 再选 style (风格)
   → 回答: "这个原子给人什么感觉?"
   → 最多 3 个, 选最准的

3. 选 audience (受众)
   → 回答: "谁最需要这个原子?"

4. 选 quality (品质)
   → 回答: "做到什么程度了?"
   → 默认 functional

5. 选 mood + use_case
   → 这两个可选, 填了加分
```

### 常见错误

```
❌ style 选太多 (>3)
❌ narrative 和 description 一模一样
❌ audience 选 "所有人" (等于没说)
❌ quality = "experimental" 又宣称 "生产环境"
❌ narrative = "test" / "测试"
```

## 四、校验规则

```
注册时自动检查:

  P0 拒绝:
    · style > 3 个
    · audience > 3 个
    · use_case > 3 个
    · narrative < 10 或 > 200 字
    · mood/quality 不在可选值列表中

  P1 警告 (星级 -0.5):
    · narrative 和 description 完全相同
    · narrative 含 "test" / "测试" / "123"
    · audience 全选 "所有人"
    · style 选 > 5 个 (如果绕过上限)

  P2 不拒绝不警告:
    · soul 完全为空 (允许, 只是 score=0)
```

## 五、Schema

```json
{
  "classification": {
    "soul": {
      "style": ["minimalist", "reliable"],
      "audience": ["backend developer", "data engineer"],
      "mood": "focused",
      "quality": "polished",
      "use_case": ["daily work", "production"],
      "narrative": "写了十年 SQL 的人, 把最趁手的工具分享出来"
    }
  }
}
```

`soul` 存在 `classification_json` 中，不需要新表、不需要新列、不需要迁移。现有 61 原子 `soul` 为空 → score=0 → 不影响现有展示。
