# 通道模型 (Channel Model)

> **概念**: 原子之间的连接线 = 活的转换器 + 血液般的通道
> **原则**: 线不是死的路径，是有类型、有逻辑、有生命的数据血管

---

## 一、线是有身份的

```
知识图谱中:

  ┌─────────┐         ┌─────────┐         ┌─────────┐
  │ http-get│ ═══[🩸]═══│json-parse│ ───[🧬]───│file-write│
  └─────────┘  直通线   └─────────┘  映射线   └─────────┘
                      │                       │
                  线把 body 映射到 text    线把 data 映射到 content
```

每根线有三个基本属性：
- **类型**：直通 / 映射 / 转换 / 合并 / 分流
- **状态**：静态(设计中) / 流动(运行中) / 阻塞(错误)
- **视觉**：颜色表示数据类型，粗细表示数据量，动画表示方向

## 二、五种线类型

### 1. 直通线 (Direct)

```
A 的输出完整传给 B 的输入，无转换

  file-read ──══[ ]══── file-write
  {content, size}  →  {path, content}

匹配规则: A 的输出字段 ⊆ B 的输入字段
```

### 2. 映射线 (Map)

```
把 A 输出的字段名映射到 B 输入的字段名

  json-parse ──[ body→text ]── string-split
  {data}          ↑            {text}
             映射: data → text

规则: 声明源字段名 → 目标字段名
```

### 3. 转换线 (Transform)

```
A 的输出类型不是 B 想要的类型，需要转换

  math-calc ──[ number→string ]── file-write
  {result: 42}    ↑                  {content: string}
             转换: toString()

规则: 声明输入类型 → 输出类型 + 转换函数
```

### 4. 合并线 (Merge)

```
多个 A 的输出合并为一个 B 的输入

  date-time ──┐
              ├──[ 合并 ]── file-write
  file-read ──┘   ↑            {content: "timestamp + data"}
              合并: 拼接/结构化

规则: 声明合并策略 (拼接 / JSON 合并 / 模板)
```

### 5. 分流线 (Split)

```
一个 A 的输出分流到多个 B

              ┌── file-write (本地存储)
  http-get ──[ 分流 ]──
              └── http-post (转发到 API)

规则: 声明分流条件 (全量 / 条件路由)
```

## 三、线的生命周期

```
[静态]           [流动]            [阻塞]           [完成]
灰色虚线 ──→ 彩色实线+动画 ──→ 红色虚线+闪烁 ──→ 绿色实线
(设计中)       (数据在流)        (报错了)         (成功)
```

## 四、画面呈现

```
知识图谱动画:

  动脉线 (出站数据):  红色, 流动方向 →, 快速动画
  静脉线 (入站数据):  蓝色, 流动方向 →, 慢速动画  
  毛管线 (字段映射):  细线, 半透明, 微动
  阻断线 (类型不匹配): 虚线, 红色, 静止

节点大小: 数据量
线粗细:   传输速率
线颜色:   数据类型 (字符串=蓝, 数字=绿, 二进制=紫, JSON=橙)
```

## 五、线也需要/meta

```
每根线在创建时也要声明自己的信息:

{
  "channel_id": "ch_001",
  "type": "map",
  "source_atom": "json-parse",
  "target_atom": "string-split",
  "mapping": {
    "data": "text"
  },
  "data_type": "string",
  "description": "把解析后的 JSON 数据传给字符串拆分"
}
```

---

## 六、为什么线是活的

```
工作流不是静态管道，是活的数据流:

  http-get 收到响应
       ↓ (数据进入动脉线)
  json-parse 解析 JSON
       ↓ (结构化数据进入映射线)
  string-split 拆分字段
       ↓ (拆分结果进入直通线)
  file-write 写入磁盘
       ↓
  [完成] 所有线变为绿色
```

线是血液——数据通过它们流动，转换，到达目的地。没有线，原子是孤岛。有了线，原子组成生命体。
